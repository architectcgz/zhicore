# Inbox 和 Outbox 分别用于什么？

## 问题

`inbox` 和 `outbox` 看起来都像“消息落库”，它们分别解决什么问题？为什么两个都可能需要？

## 回答

### 一句话区分

- `outbox` 用在消息生产侧，解决“业务数据已经提交，但 MQ 没发出去”。
- `inbox` 用在消息消费侧，解决“MQ 已经投递到了消费者，但消费者还没可靠处理完成”。

### Outbox 是什么

`outbox` 是生产者本地消息表。

典型流程是：

1. 业务事务提交时，业务数据和 `outbox_event` 一起写入同一个数据库事务。
2. 后台投递器再异步扫描 `outbox_event`。
3. 扫到待发送记录后，发送到 MQ。
4. 发送成功后，把 `outbox_event` 标记为 `DISPATCHED`。

它解决的问题是：

- 避免业务库写成功，但 MQ 发送失败，造成双写不一致。
- 让消息发送具备重试、补发、审计能力。

### Inbox 是什么

`inbox` 是消费者本地接收表。

典型流程是：

1. 消费者收到 MQ 消息。
2. 先把消息或归一化后的事件写入 `inbox`。
3. 写入成功后，这条消息才算真正进入业务系统。
4. 后续聚合、更新状态、刷新缓存都从 `inbox` 往后推进。
5. 处理完成后，把 `inbox` 状态改成 `DONE`。

它解决的问题是：

- 避免消息到达消费者后，只进了 JVM 内存或临时缓冲，进程一挂就丢。
- 给消费者侧提供幂等、重试、失败排查、断点恢复能力。

### 两者的边界

`outbox` 关注的是“怎么把消息可靠发出去”。

`inbox` 关注的是“消息到了我这里之后，怎么可靠处理完”。

所以它们不是互斥关系，而是两端各管一段：

`业务事务 -> outbox -> MQ -> inbox -> 业务处理`

### 为什么只靠 MQ 不够

因为 MQ 持久化只保证 broker 这一跳更可靠，不保证业务处理全过程可靠。

例如：

- 生产者如果先改业务数据、后发 MQ，中间失败，就需要 `outbox`。
- 消费者如果先收 MQ、后放内存缓冲，flush 前宕机，就需要 `inbox`。

### 在 zhicore 当前代码里的落点

目前已经存在 `outbox`，主要在 `content` 服务里。

- `OutboxEventPublisher` 在事务内写 `outbox_event`
- `OutboxEventDispatcher` 再把事件发到 RocketMQ

但 `ranking` 这边原来没有真正落地 `inbox`，而是：

`MQ -> consumer -> JVM 内存缓冲 -> 5 秒 flush -> Redis`

这就是当前方案的核心风险点之一，因为缓冲还没刷出去时如果进程宕机，事件就会丢。

### 为什么 ranking 方案里需要 Inbox

因为 ranking 不是简单同步写 Redis，它后面还有：

- 聚合
- 去重
- 重试
- 权威热度状态更新
- 快照刷新 Redis 榜单

这是一条多阶段链路，消息到达消费者后，必须先有一个 durable 落点，才能保证中间任何阶段失败后都能恢复。

所以对 ranking 来说，`inbox` 的意义是：

- 把“消息已到达”变成“事件已被业务系统可靠接住”
- 把“处理逻辑”从 MQ 实时回调里解耦出来

### 是否两个都必须有

不一定所有系统都同时需要，但像 zhicore 这种跨服务、异步、最终一致的链路，通常两者都值得有：

- 生产侧用 `outbox` 保证可靠发送
- 消费侧用 `inbox` 保证可靠处理

如果只有 `outbox`，消费者侧仍可能丢处理进度。

如果只有 `inbox`，生产侧仍可能出现业务提交成功但消息没发出的双写不一致。

## 最终结论

`outbox` 解决“发出去之前”的可靠性。

`inbox` 解决“收到之后”的可靠性。

两者串起来，才能把整条异步链路补完整。
