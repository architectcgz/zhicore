# 通知系统中使用 JSONB 的设计说明

## 一、背景

在传统博客或内容系统中，通知表（notifications）通常用于存储： -
评论、点赞、关注等事件通知 - 系统公告 - 小助手 / AI 提示类消息

其中往往会预留一个 `metadata`
字段，用于存放不固定、可扩展的数据。常见做法是使用 `TEXT` 类型存 JSON
字符串，但随着业务演进，这种方案会逐渐暴露问题。

------------------------------------------------------------------------

## 二、传统 TEXT 方案的主要缺点

### 1. 数据库无法理解内容结构

-   `TEXT` 只是字符串
-   数据库不知道里面是 JSON、key 是什么、值是什么
-   所有解析逻辑只能放在应用层

### 2. 几乎无法高效查询

-   无法按 JSON 内字段过滤
-   只能使用 `LIKE` / 正则，性能差且不可靠
-   随着数据量增长，查询成本迅速上升

### 3. 无法建立有效索引

-   数据库无法对字符串中的结构化字段建索引
-   通知中心的"按类型 / 来源 / 子类型筛选"会变得非常困难

### 4. schema 演进成本高

-   新增字段无法被数据库感知
-   只能靠文档或约定维护
-   容易出现字段含义混乱、命名不一致的问题

### 5. 并发更新风险

-   修改一个字段需要整段 JSON 读出再写回
-   并发更新时容易覆盖其他字段的修改

------------------------------------------------------------------------

## 三、JSONB 能解决的核心问题

### 1. 数据库可直接查询 JSON 内字段

``` sql
SELECT *
FROM notifications
WHERE metadata->>'assistant_type' = 'weekly_summary';
```

无需应用层解析，查询更直观、性能更稳定。

------------------------------------------------------------------------

### 2. 支持索引，性能可控

``` sql
CREATE INDEX idx_notifications_metadata_gin
ON notifications
USING GIN (metadata);
```

或针对具体字段：

``` sql
CREATE INDEX idx_notifications_metadata_type
ON notifications ((metadata->>'assistant_type'));
```

可以高效支撑： - 通知分类过滤 - 策略判断 - 未读统计

------------------------------------------------------------------------

### 3. 极低的 schema 演进成本

JSONB 非常适合存放： - UI 相关字段 - 策略参数 - 非核心、易变字段

新增字段无需改表结构，数据库层面依然可查询、可索引。

------------------------------------------------------------------------

### 4. 支持局部更新，减少并发冲突

``` sql
UPDATE notifications
SET metadata = jsonb_set(metadata, '{ignored}', 'true')
WHERE id = 1;
```

无需整段覆盖，更新更安全。

------------------------------------------------------------------------

### 5. 数据合法性与一致性更好

-   JSONB 只接受合法 JSON
-   自动规范化 key 顺序
-   有利于对比、缓存和去重

------------------------------------------------------------------------

## 四、JSONB 的推荐使用方式

### 1. 字段定义建议

``` sql
metadata JSONB NOT NULL DEFAULT '{}'
```

------------------------------------------------------------------------

### 2. 字段职责划分原则

**表字段中存放：** - recipient_id - type（通知大类） - is_read /
read_at - created_at

**metadata 中存放：** - 子类型（assistant_type 等） - 跳转链接（link） -
UI 渲染信息 - 策略参数 - 动作按钮定义

------------------------------------------------------------------------

### 3. metadata 示例

``` json
{
  "channel": "assistant",
  "assistant_type": "weekly_summary",
  "link": "/post/123",
  "priority": 2,
  "actions": [
    { "type": "open", "label": "查看" }
  ]
}
```

------------------------------------------------------------------------

## 五、结论

-   `TEXT` 方案简单，但数据库完全无法参与结构化处理
-   `JSONB` 在 PostgreSQL 中是成熟、稳定、性能可控的方案
-   对于通知 / 小助手这类 **形态多、演进快、查询逐渐复杂** 的业务，JSONB
    是明显更优的选择

> 建议：核心字段关系型化，变化快的扩展字段 JSONB 化。
