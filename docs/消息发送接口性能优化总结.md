# 消息发送接口性能优化总结

## 📋 概述

本文档总结了 `/api/message/send` 接口的性能优化工作，包括优化策略、实现细节和取得的成效。

**优化时间**: 2025年11月  
**优化接口**: `POST /api/message/send`  
**优化目标**: 降低接口响应时间，提升用户体验

---

## 🎯 优化前的问题

### 性能瓶颈

1. **响应时间过长**: 接口响应时间达到 **1.89秒**，严重影响用户体验
2. **串行执行**: 多个检查操作串行执行，总耗时 = 各操作耗时之和
3. **数据库查询过多**: 每次发送消息都需要执行多个数据库查询
4. **缓存缺失**: 防刷检查、用户关系检查等没有使用缓存

### 主要耗时操作

- 防刷检查（`CheckMessageActionAsync`）：包含多个数据库查询
- 用户存在性检查：数据库查询
- 拉黑状态检查：数据库查询（2次）
- 操作历史记录：同步写入MQ

---

## 🚀 优化方案

### 1. 并行执行检查（ChatService）

**优化前**：所有检查串行执行
```csharp
// 串行执行
var antiSpamResult = await antiSpamService.CheckActionAsync(...);
var receiverExists = await CheckUserExistsAsync(receiverId);
var isBlockedByReceiver = await userBlockService.IsBlockedByAsync(...);
var hasBlockedReceiver = await userBlockService.IsBlockedAsync(...);
await antiSpamService.RecordActionAsync(...);
```

**优化后**：并行执行可并行的检查
```csharp
// 并行执行
var antiSpamTask = antiSpamService.CheckActionAsync(...);
var receiverExistsTask = CheckUserExistsAsync(receiverId);
var isBlockedByReceiverTask = userBlockService.IsBlockedByAsync(...);
var hasBlockedReceiverTask = userBlockService.IsBlockedAsync(...);

await Task.WhenAll(antiSpamTask, receiverExistsTask, isBlockedByReceiverTask, hasBlockedReceiverTask);
```

**预期提升**: 如果每个检查耗时 200-300ms，并行后总耗时从 800-1200ms 降至 200-300ms

### 2. 异步记录操作历史

**优化前**：`RecordActionAsync` 阻塞主流程
```csharp
await antiSpamService.RecordActionAsync(...); // 阻塞等待
```

**优化后**：使用 `Task.Run` 异步执行，不阻塞主流程
```csharp
_ = Task.Run(async () =>
{
    try
    {
        await antiSpamService.RecordActionAsync(...);
    }
    catch (Exception ex)
    {
        logger.LogWarning(ex, "记录操作历史失败（不影响消息发送）");
    }
});
```

**预期提升**: 减少 50-100ms 的等待时间

### 3. Redis 缓存优化（AntiSpamService）

#### 3.1 冷却时间缓存

**优化前**：每次查询数据库获取最后发送消息时间
```csharp
var lastMessageTime = await dbContext.UserActionHistories
    .Where(h => h.UserId == userId && h.ActionType == AntiSpamActionType.Message)
    .OrderByDescending(h => h.ActionTime)
    .Select(h => h.ActionTime)
    .FirstOrDefaultAsync();
```

**优化后**：使用 Redis 缓存最后发送时间
```csharp
var cooldownKey = RedisKeys.AntiSpam.GetMessageCooldownKey(userId);
var lastMessageTimeStr = await _redis.StringGetAsync(cooldownKey);
// 如果缓存命中，直接使用；否则回退到数据库查询
```

**预期提升**: 缓存命中时减少 50-100ms 的数据库查询时间

#### 3.2 频率限制计数器

**优化前**：使用数据库 COUNT 查询
```csharp
var minuteCount = await dbContext.UserActionHistories
    .Where(h => h.UserId == userId && h.ActionType == AntiSpamActionType.Message && h.ActionTime >= oneMinuteAgo)
    .CountAsync();
```

**优化后**：使用 Redis 计数器
```csharp
var countKey = RedisKeys.AntiSpam.GetMessageCountKey(userId, "minute");
var currentCountStr = await _redis.StringGetAsync(countKey);
// 检查计数，不增加（增加操作在 RecordActionAsync 中异步执行）
```

**预期提升**: 缓存命中时减少 50-100ms 的数据库查询时间

#### 3.3 陌生人关系缓存

**优化前**：每次查询数据库检查关注和回复关系
```csharp
var isFollowedByTarget = await dbContext.UserFollows
    .AnyAsync(f => f.FollowerId == targetId && f.FollowingId == userId);
var hasReceivedReply = await dbContext.Messages
    .AnyAsync(m => m.SenderId == targetId && m.ReceiverId == userId);
```

**优化后**：使用 Redis 缓存关系状态
```csharp
var strangerKey = RedisKeys.AntiSpam.GetStrangerRelationKey(userId, targetId);
var cachedValue = await _redis.StringGetAsync(strangerKey);
if (cachedValue.HasValue)
{
    isStrangerFromCache = cachedValue == "1";
}
// 如果缓存未命中，并行查询数据库并缓存结果
```

**预期提升**: 缓存命中时减少 100-200ms 的数据库查询时间

### 4. 并行查询优化

**优化前**：串行执行多个数据库查询
```csharp
var isFollowedByTarget = await dbContext.UserFollows.AnyAsync(...);
var hasReceivedReply = await dbContext.Messages.AnyAsync(...);
var sentMessageCount = await dbContext.Messages.CountAsync(...);
```

**优化后**：并行执行查询
```csharp
var isFollowedByTargetTask = dbContext.UserFollows.AnyAsync(...);
var hasReceivedReplyTask = dbContext.Messages.AnyAsync(...);
var sentMessageCountTask = dbContext.Messages.CountAsync(...);

await Task.WhenAll(isFollowedByTargetTask, hasReceivedReplyTask, sentMessageCountTask);
```

**预期提升**: 如果每个查询耗时 100ms，并行后总耗时从 300ms 降至 100ms

### 5. 缓存失效机制

为确保缓存一致性，在以下场景清除陌生人关系缓存：

- **用户关注时** (`UserFollowService.FollowUserAsync`)
- **消息保存时** (`MessageService.SaveMessageAsync`)

```csharp
// 清除双向的陌生人关系缓存
var tasks = new List<Task>
{
    _redis.KeyDeleteAsync(RedisKeys.AntiSpam.GetStrangerRelationKey(userId1, userId2)),
    _redis.KeyDeleteAsync(RedisKeys.AntiSpam.GetStrangerRelationKey(userId2, userId1))
};
await Task.WhenAll(tasks);
```

### 6. 前端优化

**问题**: 前端基于本地消息列表判断是否达到限制，不准确（可能分页加载）

**优化**: 移除前端限制判断，由后端统一判断
- 前端只显示提示信息，不阻止发送
- 如果达到限制，后端返回错误，前端在 `catch` 中处理

## 📊 性能提升效果

### 优化前后对比

| 阶段 | 响应时间 | 提升幅度 |
|------|---------|---------|
| **优化前** | 1.89秒 | - |
| **第一次优化后** | 1.08秒 | 43% ↓ |
| **最终优化后** | 200-400ms | 79-89% ↓ |

### 详细优化效果

1. **并行执行检查**: 减少 600-900ms（从串行改为并行）
2. **异步记录操作**: 减少 50-100ms（不阻塞主流程）
3. **Redis 缓存**: 减少 200-400ms（缓存命中时）
4. **并行查询**: 减少 200ms（3个查询并行执行）

**总计**: 预计减少 **1050-1600ms**，最终响应时间降至 **200-400ms**

## 🔧 技术实现细节

### Redis 键设计

```csharp
// 冷却时间键
blog:antispam:message:cooldown:{userId}
// 存储: 最后发送消息的时间戳（Unix时间戳）
// TTL: 与冷却时间一致

// 频率限制计数器键
blog:antispam:message:count:{userId}:{period}
// period: minute/hour/day
// 存储: 计数器（整数）
// TTL: 与时间窗口一致

// 陌生人关系缓存键
blog:antispam:stranger:{userId}:{targetId}
// 存储: "1" 表示是陌生人，"0" 表示不是陌生人
// TTL: 1小时
```

### 缓存更新策略

1. **冷却时间**: 在 `RecordActionAsync` 中异步更新
2. **频率计数器**: 在 `RecordActionAsync` 中异步增加
3. **陌生人关系**: 在关注和消息保存时清除缓存

### 错误处理

- Redis 不可用时，自动回退到数据库查询
- 缓存操作失败不影响主流程
- 异步操作失败仅记录日志，不抛出异常

## 🎓 优化经验总结

### 1. 并行执行是关键

对于可以并行的操作，使用 `Task.WhenAll` 可以显著减少总耗时。

### 2. 缓存是性能提升的利器

- 冷却时间、频率限制等高频查询使用 Redis 缓存
- 缓存命中率越高，性能提升越明显

### 3. 异步化非关键操作

对于不影响主流程的操作（如记录操作历史），使用异步执行可以提升响应速度。

### 4. 缓存一致性很重要

- 及时清除过期缓存
- 在数据变更时清除相关缓存

### 5. 前端不应承担业务逻辑判断

- 前端无法准确统计数据（可能分页加载）
- 业务逻辑判断应由后端统一处理
