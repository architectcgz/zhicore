# Blog 标签（Tag）功能需求文档

## 1. 背景与目标

随着博客内容数量的增长，需要一种**结构化的内容分类与检索机制**，以支持：

* 用户通过标签（Tag）快速发现相关内容
* 为搜索、推荐、聚合页面提供基础数据
* 为后续运营（热门标签、标签管理、SEO）预留扩展空间

因此，本项目决定在现有 Blog 系统中**引入独立的 Tag 模型**，并建立 Post 与 Tag 的多对多关系。

---

## 2. 需求范围（Scope）

### 2.1 本期需求（必须支持）

1. 支持为 Post 绑定多个 Tag
2. 支持通过 Tag 查询包含该 Tag 的 Post 列表
3. 支持以 Tag 作为独立实体进行展示（Tag 标题 / slug）
4. 保证 Tag 在系统内的唯一性与规范化
5. 支持后期扩展统计、搜索、管理能力

### 2.2 明确不在本期范围内（但需可扩展）

* Tag 的层级关系（父子 Tag）
* Tag 权重 / 推荐算法
* Tag 订阅 / 关注功能
* Tag 的国际化（i18n）

---

## 3. 业务概念定义

### 3.1 Tag

Tag 是对 Post 内容主题的**轻量级、可复用标识**，具有以下特征：

* 全局唯一（通过 slug 保证）
* 可被搜索、展示和统计
* 可关联多个 Post

### 3.2 Post 与 Tag 的关系

* 一个 Post 可以关联 0~N 个 Tag
* 一个 Tag 可以关联 0~N 个 Post
* 关系为标准 **多对多（Many-to-Many）**

---

## 4. 功能性需求

### 4.1 Tag 管理（系统层）

1. 系统内每个 Tag 必须有唯一的 `slug`
2. Tag 的展示名称（name）与 slug 分离
3. Tag 不允许因大小写或空格差异重复创建
4. 支持通过 slug 精确定位 Tag

> 示例：
>
> * name: "PostgreSQL"
> * slug: "postgresql"

### 4.2 Post 创建 / 编辑时的 Tag 行为

1. 创建或编辑 Post 时，允许传入 Tag 列表
2. 系统应对 Tag 做规范化处理（slug 化）
3. 不存在的 Tag 可自动创建（由系统控制）
4. Post 与 Tag 的关联需防止重复绑定
5. 单个 Post 的 Tag 数量应有上限（建议 ≤ 10）

### 4.3 按 Tag 查询 Post

1. 支持通过 Tag slug 查询 Post 列表
2. 查询结果支持分页与排序（如按发布时间倒序）
3. 查询性能需随 Post 数量线性扩展

典型场景：

* Tag 页面：`/tags/{slug}`
* 搜索过滤：`tag = postgresql`

### 4.4 Tag 列表与统计（预留）

系统应支持后期扩展以下能力：

* 获取 Tag 列表（分页）
* 统计每个 Tag 下的 Post 数量
* 按热度排序 Tag

---

## 5. 非功能性需求

### 5.1 性能要求

* 按 Tag 查询 Post 应使用索引，避免全表扫描
* Tag-Post 关系表可线性增长，但查询性能需可预测

### 5.2 数据一致性

* Tag 的权威数据源为 PostgreSQL
* Post 与 Tag 的关系需具备唯一性约束
* 系统应避免出现语义重复的 Tag（如 js / JS / JavaScript）

### 5.3 可扩展性

设计需支持未来以下演进而不破坏现有数据模型：

* Tag 描述、颜色、封面图
* Tag 别名 / 合并
* SEO 优化（slug 不变，展示名可变）
* 读模型（如 MongoDB / 搜索引擎）冗余

---

## 6. 数据模型设计原则（约束性需求）

1. Tag 必须作为**独立实体表**存在
2. Post 与 Tag 的关系必须使用关联表建模
3. 关联表需保证：

   * 不可重复绑定
   * 支持高效的双向查询
4. 禁止将 Tag 仅作为非结构化字段（如 JSON / text 拼接）存储在 Post 表中

---

## 7. 技术选型约束

* PostgreSQL 作为 Tag 与 Post-Tag 关系的权威存储
* MongoDB 可作为读模型冗余 Tag 数据（非强制）
* Tag 的唯一性以 `slug` 为准，而非展示名

---

## 8. 典型使用场景示例

### 场景 1：用户访问某个 Tag 页面

* 输入：Tag slug
* 输出：该 Tag 下的 Post 列表（分页、排序）

### 场景 2：发布一篇带 Tag 的 Post

* 输入：Post 内容 + Tag 列表
* 行为：

  * 规范化 Tag
  * 创建缺失 Tag
  * 建立 Post-Tag 关联

---

## 9. 设计原则总结

* **Tag 是实体，不是字符串**
* **关系可以变大，但必须可控、可索引**
* **写入稍复杂，换取查询与演进的长期稳定性**

该需求文档将作为后续数据库设计、接口设计与实现的重要依据。
