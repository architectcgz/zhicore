# 模块设计：草稿管理

## 1. 模块目标
草稿模块用于支撑编辑器“自动保存/手动保存/恢复草稿”能力，核心目标：
- 降低用户编辑内容丢失风险。
- 支持按用户、按文章快速恢复最近编辑状态。
- 在高频写场景下保持可接受的写入与读取延迟。

## 2. 功能边界
### 2.1 对外接口
- `POST /api/v1/posts/{postId}/draft`：保存草稿（自动/手动）。
- `GET /api/v1/posts/{postId}/draft`：获取某篇文章最新草稿。
- `GET /api/v1/posts/drafts`：获取当前用户全部草稿。
- `DELETE /api/v1/posts/{postId}/draft`：删除草稿。

### 2.2 核心组件
- 门面与编排：`PostController`、`PostFacadeService`、`PostApplicationService`
- 领域端口：`DraftService`
- 基础实现：`DraftServiceImpl`（Mongo 持久化）
- 缓存装饰器：`CachedDraftService`（Redis Cache-Aside）
- 数据模型：`PostDraft`、`PostDraftRepository`

## 3. 数据与存储设计
### 3.1 Mongo 文档 `post_drafts`
- 关键字段：`postId`、`userId`、`content`、`contentType`、`savedAt`、`isAutoSave`、`wordCount`
- 唯一索引：`(postId, userId)`，保证“一篇文章一个用户仅一份草稿”
- 查询索引：`(userId, savedAt desc)`，支持用户草稿列表
- TTL 索引：`savedAt`（当前固定 30 天）

### 3.2 Redis 缓存键
- 单草稿：`post:draft:{postId}:{userId}`
- 用户草稿列表：`post:drafts:user:{userId}`
- 策略：命中返回，未命中回源 Mongo，回填缓存；写操作后删缓存。

## 4. 核心流程
### 4.1 保存草稿
1. `PostApplicationService.saveDraft` 校验文章存在与归属。
2. 调用 `DraftService.saveDraft` 进行 upsert。
3. `DraftServiceImpl` 计算字数、更新 `savedAt` 并写 Mongo。
4. `CachedDraftService` 删除相关缓存键，保证后续读到新值。

### 4.2 查询草稿
1. `CachedDraftService` 先查 Redis。
2. miss 时回源 `DraftServiceImpl` 查询 Mongo。
3. 命中则写回缓存；空结果写空值缓存防穿透。

### 4.3 删除草稿
1. 校验文章归属。
2. 删除 Mongo 草稿记录。
3. 失效单草稿与用户列表缓存。

## 5. 一致性模型
- 主数据源：Mongo（`post_drafts`）。
- Redis 为性能层，允许短暂不一致。
- 失败策略当前偏“可用性优先”：草稿写失败不抛错阻断主流程。

## 6. 当前实现问题（需优化）
1. `SaveDraftRequest.contentType/deviceId` 在应用层未透传，`DraftServiceImpl` 固定写 `MARKDOWN`，`deviceId` 丢失。
2. `DraftServiceImpl.saveDraft` 捕获异常后仅记录日志，不抛出；上层仍记录“保存成功”，存在静默失败风险。
3. `post.draft.max-drafts-per-user` 配置未实际生效，当前没有用户草稿数量上限控制。
4. `cleanExpiredDrafts` 方法存在但未看到调度入口，过期清理主要依赖 Mongo TTL。
5. `DraftProperties.expirationDays` 与 `PostDraft` 上硬编码 TTL（30 天）可能配置漂移。

## 7. 优化建议
1. 完整透传 `contentType`、`deviceId`，并做合法值校验与落库。
2. 草稿保存改为“可观测失败”：至少返回可识别状态或事件，避免前端误判成功。
3. 增加 `maxDraftsPerUser` 执行策略（插入前裁剪最旧草稿）。
4. 统一过期策略来源，避免“代码注解 TTL”与“配置 TTL”双轨。
5. 增加草稿关键指标：保存成功率、平均耗时、缓存命中率、静默失败计数。
