### 2026-02-25：创建文章接口返回 400（阻塞多数 API 集成测试）

- 影响范围：`POST /api/v1/posts`（创建草稿）失败，导致依赖“先创建文章再更新/发布/点赞/收藏/分页查询”的集成测试无法继续推进。
- 复现方式：
  - 执行（PowerShell 建议使用 `--%` 避免逗号等被解析）：
    - `mvn --% -pl zhicore-content -am -Dtest=PostControllerCrudIntegrationTest#createUpdatePublishAndList -Dsurefire.failIfNoSpecifiedTests=false test`
  - 现象：创建文章请求返回 HTTP 400，响应体类似：`{\"code\":1001,\"message\":\"参数值无效\",...}`
- 关键日志/根因定位：
  - 全局异常处理器打印：`aggregateVersion 不能为空（Outbox 事件必须携带聚合版本号）`
  - 说明：创建草稿链路在写 Outbox 的集成事件时，`aggregateVersion` 为 `null`，被 `OutboxEventPublisher` 的强校验拦截，最终映射为参数错误（1001）。
- 结论：这不是测试问题，而是“创建文章主链路”的真实缺陷；需要保证创建文章产生的 Outbox 集成事件携带非空 `aggregateVersion`（建议取数据库最新版本号语义）。
- 状态：待修复（修复后需回归：创建/更新/发布/查询相关用例）。

### 2026-02-25：定时发布扫描的 dbNow 时间类型转换异常

- 影响范围：调度扫描相关集成测试（`ScheduledPublishScannerIntegrationTest`）。
- 复现方式：
  - `mvn -pl zhicore-content -am -Dtest=ScheduledPublishScannerIntegrationTest -Dsurefire.failIfNoSpecifiedTests=false test`
- 现象：调用 `scheduledPublishEventRepository.dbNow()` 时抛出异常：`Cannot convert the column of type TIMESTAMPTZ to requested type java.time.LocalDateTime`
- 根因：
  - Mapper 使用 `SELECT CURRENT_TIMESTAMP`（PostgreSQL 返回 `TIMESTAMPTZ`），但方法返回类型为 `LocalDateTime`，导致类型不匹配。
- 修复建议：
  - 改为 `SELECT LOCALTIMESTAMP` 或 `SELECT CURRENT_TIMESTAMP::timestamp`，确保返回无时区的 `timestamp`，与 `LocalDateTime` 对齐。
- 状态：待修复（修复后需回归扫描重入队/到点发布等路径）。

### 2026-02-25：参数校验错误码约定（测试断言修正项）

- 现象：接口返回 HTTP 400 时，`ApiResponse.code` 并非 400，而是业务约定的 `ResultCode.PARAM_ERROR=1001`。
- 结论：集成测试断言应以“HTTP 状态码 + 业务 code（1001）”为准，避免把 `ApiResponse.code` 误当作 HTTP 状态码。

### 2026-02-25：修复记录与回归结论

- 已修复：创建文章写 Outbox 时 `aggregateVersion=null` 导致创建接口 400
  - 修复点：创建草稿流程在发出创建事件前，显式取数据库最新版本号并写入事件（避免内存聚合未回写 version）。
  - 回归：`PostControllerCrudIntegrationTest#createUpdatePublishAndList` 通过；`mvn -pl zhicore-content -am test` 通过。
- 已修复：`scheduledPublishEventRepository.dbNow()` 的 TIMESTAMPTZ → LocalDateTime 转换异常
  - 修复点：`SELECT CURRENT_TIMESTAMP` 改为 `SELECT LOCALTIMESTAMP`。
  - 回归：`ScheduledPublishScannerIntegrationTest` 通过。
- 已修复：已发布列表游标分页 SQL 在 cursor 为空时触发 PostgreSQL 参数类型推断错误
  - 现象：`ERROR: could not determine data type of parameter $1`（`selectPublishedCursor`）。
  - 修复点：使用 MyBatis 动态 SQL（cursor 为空时不拼接游标条件），避免 NULL 参数参与预编译类型推断。
  - 回归：`PostControllerCrudIntegrationTest#createUpdatePublishAndList` 中的列表查询路径通过。
- 已修复：补扫重置闸门时 `last_enqueue_at` 置空未落库
  - 根因：MyBatis-Plus 默认忽略 NULL 值更新，导致 `updateById` 无法把 `last_enqueue_at`/`last_error` 更新为 NULL。
  - 修复点：对 `ScheduledPublishEventEntity.lastEnqueueAt/lastError` 启用允许 NULL 更新的策略。
  - 回归：`ScheduledPublishScannerIntegrationTest#resetStaleGateShouldClearLastEnqueueAt` 通过。
