# zhicore-content 总体架构设计（接手版）

## 1. 文档目标
本文件面向“新接手该服务的研发同学”，目标是回答以下问题：
- 这个服务管什么，不管什么。
- 请求从接口进来后，如何穿过各层到存储和消息系统。
- 数据放在哪里，为什么这么放。
- 出问题时先看哪里、怎么定位。

## 2. 业务定位与边界
### 2.1 服务职责
`zhicore-content` 负责内容域核心能力：
- Post 的生命周期：创建草稿、更新、发布、撤回、定时发布、删除、恢复。
- Tag 的查询与绑定关系维护。
- 用户互动：点赞、收藏及计数。
- 集成事件输出：通过 Outbox 向 RocketMQ 可靠投递事件。

### 2.2 非职责（由其他服务负责）
- 用户主数据（昵称/头像来源）：`user-service`
- 文件元数据与 URL 生成：`upload-service`
- 分布式 ID：`id-generator`

### 2.3 上下游依赖
- 输入：网关转发的 HTTP 请求、MQ 消费消息（定时发布等）。
- 输出：PostgreSQL、MongoDB、Redis、RocketMQ。

## 3. 分层与包结构
### 3.1 Interface 层（`interfaces`）
- 主要类：
  - `PostController`
  - `TagController`
  - `PostLikeController`
  - `PostFavoriteController`
  - `AdminPostController`
- 职责：协议与参数适配、认证上下文获取、返回统一 `ApiResponse`。

### 3.2 Application 层（`application`）
- 门面：`PostFacadeService`
- 核心编排：`PostApplicationService`、`TagApplicationService`
- 工作流：`CreateDraftWorkflow`、`PublishPostWorkflow`
- 命令处理：`application/command/handlers/*`
- 查询模型：`PostQueryServiceImpl` + 缓存装饰器 `CacheAsidePostQuery`
- 职责：用例编排、事务边界、跨仓储/跨依赖协同、事件转换与发布。

### 3.3 Domain 层（`domain`）
- 聚合/实体：`Post`、`Tag`、`PostLike`、`PostFavorite`
- 值对象：`PostId`、`UserId`、`TagId`、`TopicId`
- 领域事件：`PostCreatedDomainEvent`、`PostPublishedDomainEvent` 等
- 职责：业务规则与状态迁移，不关心技术细节。

### 3.4 Infrastructure 层（`infrastructure`）
- PG 持久化：`infrastructure/persistence/pg/*`
- Mongo 持久化：`infrastructure/persistence/mongo/*`
- 缓存/锁：`infrastructure/cache/*`
- 消息：`OutboxEventPublisher`、`OutboxEventDispatcher`
- 职责：技术实现与外部系统集成。

## 4. 存储模型与职责分工
### 4.1 PostgreSQL（强一致元数据）
- `posts`：文章主元数据（状态、作者快照、版本、发布时间、写入状态）
- `post_tags` / `tags`：标签关系
- `post_stats` / `tag_stats`：统计计数
- `outbox_event`：待投递集成事件
- `consumed_events`：消费幂等记录

### 4.2 MongoDB（内容与草稿）
- 文章正文、草稿快照等大文本/高变数据。
- 目标：避免在 PG 中承载高频大字段写入。

### 4.3 Redis（性能层）
- 文章详情缓存、列表缓存、热门标签缓存
- 点赞/收藏状态与计数缓存
- 分布式锁（缓存击穿、调度并发互斥）

## 5. 关键调用链（按典型场景）
### 5.1 创建草稿
`PostController -> PostFacadeService -> PostApplicationService#createPost -> CreateDraftWorkflow -> PG + Mongo -> 领域事件 -> Outbox`

### 5.2 发布文章
`PostController -> PostApplicationService#publishPost -> PublishPostWorkflow -> Post状态变更 -> Outbox(PostPublished)`

### 5.3 详情查询
`PostController -> PostApplicationService#getPostById -> CacheAsidePostQuery -> PostQueryServiceImpl -> (PG + Mongo + stats)`

### 5.4 点赞/收藏
`Controller -> ApplicationService -> 事务内写库 -> 事务后写Redis -> 发布互动事件`

### 5.5 Outbox 投递
`业务事务写 outbox_event(PENDING) -> Dispatcher 扫描 -> RocketMQ 投递 -> 更新 DISPATCHED/FAILED`

## 6. 一致性策略（必须理解）
### 6.1 单服务内
- 以 PG 事务为核心提交单元。
- 应用服务是主要事务边界。

### 6.2 跨存储（PG + Mongo）
- 非分布式事务；使用 workflow 的顺序写入 + best-effort 清理。
- 结果：允许极小窗口的临时不一致，需要巡检与补偿能力。

### 6.3 跨服务事件
- 采用 Outbox，避免“业务写库成功但 MQ 发送失败”的双写问题。
- 最终一致：依赖 Dispatcher 重试与下游幂等。

## 7. 运行期关键状态
### 7.1 Post 业务状态
- `DRAFT` -> `PUBLISHED`
- `DRAFT/PUBLISHED` -> `SCHEDULED` -> `PUBLISHED`
- 软删除进入 `DELETED`，可恢复

### 7.2 WriteState（写入阶段）
- 用于识别元数据写入与内容写入阶段，帮助保障发布前校验。

## 8. 可观测性与排障入口
### 8.1 先看什么
1. 应用日志（按 `postId`、`eventId`、`userId` 检索）
2. Outbox 表状态分布（PENDING/FAILED）
3. Redis 命中率与异常计数
4. MQ 消费堆积与重试

### 8.2 常见告警建议
- Outbox `PENDING` 长时间积压
- Outbox `FAILED` 增长
- 点赞/收藏计数与明细偏差超阈值
- Mongo 读失败率上升导致 `contentDegraded` 比例升高

## 9. 接手优先阅读顺序
1. `PostController`、`PostFacadeService`
2. `PostApplicationService`（最大编排中心）
3. `CreateDraftWorkflow`、`PublishPostWorkflow`
4. `CacheAsidePostQuery`、`PostQueryServiceImpl`
5. `OutboxEventPublisher`、`OutboxEventDispatcher`
6. `src/main/resources/db/schema.sql`

## 10. 当前设计缺陷（简表）
- `GET /api/v1/posts` 的 `sort/status` 参数未真正下传执行。
- `PostStatus.valueOf(status)` 缺少输入保护。
- Redis 使用 `keys(pattern)` 的点位仍存在。
- 点赞/收藏计数递减缺乏下界保护。
- Outbox 失败告警未真正接通。
