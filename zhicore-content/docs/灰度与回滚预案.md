# 灰度与回滚预案（T22）

## 背景

架构修复（R1~R18）涉及定时发布、Outbox、缓存键结构与失效策略、分页规则与并发冲突语义等改动。为降低发布风险，需提供可执行的灰度与回滚步骤。

## 灰度步骤（建议）

1. **预检查**
   - 确认数据库 DDL 已生效（`scheduled_publish_event`、`post_likes`、`post_favorites`、`outbox_retry_audit`）。
   - 确认监控指标可采集（Outbox 堆积/失败、数据库耗时等）。
2. **小流量灰度**
   - 先灰度到单实例/少量实例，观察 30~60 分钟：
     - Outbox `PENDING` 是否堆积、最老年龄是否上升；
     - 定时发布是否按 DB 时间门禁执行；
     - 列表分页是否出现重复/遗漏；
     - 缓存失效是否出现扫描异常与性能抖动。
3. **逐步扩大**
   - 若关键指标稳定，逐步扩容到全量。

## 回滚策略

### 1) 代码回滚

- 若出现不可接受的错误，优先回滚到上一稳定版本。
- 回滚前确认：新引入的表/字段不会破坏旧版本读写（仅增量表/字段一般可向后兼容）。

### 2) 缓存兼容窗口

- 列表缓存 key 带版本前缀（例如 `v2`），结构升级可通过 bump 版本避免旧缓存污染。
- 回滚时建议保留兼容窗口：回滚后清理新版本列表 key（pattern 删除或按前缀清理）。

### 3) 人工介入兜底

- Outbox 失败事件可通过管理端接口人工重试：
  - `GET /api/v1/admin/outbox/failed`
  - `POST /api/v1/admin/outbox/{eventId}/retry`
- 定时发布异常可通过扫描/补扫机制收敛；若出现系统性失败需结合告警与日志排查。
