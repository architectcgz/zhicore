# 模块设计：调度任务与分布式锁

## 1. 模块目标
该模块用于保障后台任务在多实例部署下的互斥执行与可恢复运行，主要目标：
- 避免同一任务被多个实例并发执行。
- 在任务执行时间不确定时避免锁过期导致重复执行。
- 为定时清理、回填、Outbox 投递提供统一锁能力。

## 2. 任务清单与锁策略
### 2.1 使用分布式锁的任务
- `OutboxEventDispatcher.dispatch`
  - 锁：`lockKeys.outboxDispatcher()`
  - 策略：`tryLockWithWatchdog`（自动续期）
- `IncompletePostCleanupScheduler.cleanupIncompletePosts`
  - 锁：`lockKeys.incompletePostCleanup()`
  - 策略：固定 TTL（5 分钟）
- `ConsumedEventCleanupScheduler.cleanupOldEvents`
  - 锁：`lockKeys.consumedEventCleanup()`
  - 策略：固定 TTL（5 分钟）
- `AuthorInfoBackfillScheduler.backfillUnknownAuthors`
  - 锁：`lockKeys.authorInfoBackfill()`
  - 策略：`tryLockWithWatchdog`

### 2.2 未使用分布式锁的调度
- `MetricsMonitorScheduler`（指标检查与日志输出）

## 3. 锁能力设计
### 3.1 接口
- `LockManager`
  - `tryLock(key, wait, lease)`：固定过期时间
  - `tryLockWithWatchdog(key, wait)`：启用 Redisson 看门狗
  - `unlock(key)`：仅允许当前线程持有者释放

### 3.2 键命名
- `LockKeys` 统一生成：`{env}:{service}:{category}:{resource}:{action}:lock`
- `env` 优先取 `app.env`，否则取 `spring.profiles.active` 首 profile

## 4. 任务执行时序（通用）
1. 计算任务锁 key。
2. `tryLock...` 获取锁，失败即跳过本轮。
3. 执行业务逻辑（清理、回填、投递等）。
4. `finally` 中释放锁。

## 5. 当前实现问题（需优化）
1. `IncompletePostCleanupScheduler`、`ConsumedEventCleanupScheduler` 使用固定 5 分钟 lease，长任务场景下存在锁提前过期风险。
2. 多个重任务集中在凌晨 2 点（消费记录清理、作者回填等），峰值时段资源竞争明显。
3. 锁获取失败仅日志记录，缺少任务“连续跳过次数”监控告警，容易长期无感知失效。
4. `lockManager.unlock` 在锁已过期/非当前线程场景仅告警，未提供任务补偿或自检机制。
5. 任务执行缺少统一的超时、分批进度与中断恢复协议，复杂任务可运维性一般。

## 6. 优化建议
1. 不确定耗时任务统一改为 `watchdog`，固定 TTL 仅用于短、可预测任务。
2. 错峰调度：将 2 点重任务拆散到不同时间窗，降低 DB/Redis/MQ 峰值冲击。
3. 增加任务治理指标：锁获取成功率、跳过次数、执行耗时、失败率、处理条数。
4. 引入任务“心跳/进度”记录，支持运行中可观测和异常恢复。
5. 对关键清理任务补充幂等标记与重入保护，确保锁异常时也不产生破坏性副作用。
