# 模块设计：事件消费与幂等

## 1. 模块目标
该模块负责消费外部集成事件并更新内容域读写模型，核心目标：
- 在“至少一次投递”语义下避免重复副作用。
- 在消息乱序、重复投递、重启恢复场景下保持可预期结果。
- 将消费状态沉淀到可清理的持久化介质。

## 2. 消费者与职责
- `StatsUpdatedConsumer`：消费文章统计覆盖更新事件。
- `UserProfileUpdatedConsumer`：消费用户资料更新并同步作者快照。
- `UserProfileChangedConsumer`：兼容旧 topic 的用户资料同步链路。
- `PostScheduleConsumer`：消费定时发布执行事件。
- `AuthorInfoCompensationConsumer`：消费作者信息补偿事件。

## 3. 幂等基础设施
### 3.1 `consumed_events` 表
- 字段：`event_id`、`event_type`、`consumer_name`、`consumed_at`
- 当前主键：`event_id`

### 3.2 仓储接口
- `ConsumedEventRepository.tryInsert(eventId, eventType, consumerName)`
- 语义：插入成功视为首次消费，主键冲突视为重复消息。

## 4. 典型消费流程（有幂等表场景）
1. 反序列化消息。
2. `tryInsert` 抢占消费记录。
3. 插入失败则判定重复消息并返回。
4. 执行业务更新（统计 upsert、作者快照同步等）。
5. 失效相关缓存。

## 5. 不同消费者的一致性策略
### 5.1 `StatsUpdatedConsumer`
- 使用 `consumed_events` + 统计覆盖写（upsert）双保险。
- 覆盖写天然可重放，重复事件副作用较低。

### 5.2 `UserProfileUpdatedConsumer`
- 使用 `consumed_events` 去重。
- 再叠加 `owner_profile_version` 版本过滤抵御乱序。

### 5.3 `PostScheduleConsumer`
- 未接入 `consumed_events`。
- 依赖 `post.status == SCHEDULED` 判定是否执行，状态机幂等。

### 5.4 `AuthorInfoCompensationConsumer`
- 未接入 `consumed_events`。
- 依赖 `hasDefaultAuthorInfo()` 做业务幂等。

## 6. 当前实现问题（需优化）
1. `consumed_events` 仅以 `event_id` 为主键，`consumer_name` 未参与唯一约束；多消费者共享同一 `event_id` 时会相互干扰。
2. `ConsumedEventRepositoryImpl.tryInsert` 捕获通用异常后返回 `false`，会把“存储故障”误判为“重复消息”，存在消息静默丢弃风险。
3. 并非所有消费者都接入统一去重机制（如定时发布、补偿），幂等策略分散在业务判断，治理成本高。
4. 存在两套用户资料事件消费者（新旧 topic 并存），容易出现语义分叉和重复维护。
5. 缺少统一死信/重放治理规范（失败消息如何补偿、如何人工重放）在代码层尚不完整。

## 7. 优化建议
1. 将幂等主键升级为 `(event_id, consumer_name)` 复合唯一键，避免跨消费者冲突。
2. `tryInsert` 遇到非重复异常时应抛出异常触发 MQ 重试，而不是返回 `false`。
3. 制定消费者幂等分级标准：强制接入幂等表，业务判定仅作二次保护。
4. 合并用户资料更新消费链路，统一消息契约与版本控制。
5. 建立“失败重放”能力：死信队列 + 管理端重放工具 + 运行手册。
