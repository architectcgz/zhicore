# 模块设计：点赞与收藏（接手版）

## 1. 模块目标
支持用户对已发布文章执行点赞/收藏与取消操作，并提供状态查询与计数查询，保障高并发下的正确性与性能。

## 2. 核心类
- `PostLikeApplicationService`
- `PostFavoriteApplicationService`
- 对应控制器：`PostLikeController`、`PostFavoriteController`

## 3. 流程（点赞为例，收藏同构）
1. 先查 Redis 用户状态 key（快速失败重复点赞）。
2. 校验文章存在且状态为 `PUBLISHED`。
3. 事务内写库：
   - DB 再校验一次（防并发）
   - 生成 interaction id
   - 插入明细表
4. 事务提交后更新 Redis：
   - `likeCount` 自增
   - 用户状态 key 写入
5. 发布 `PostLikedIntegrationEvent`（Outbox）。

取消流程反向执行：删除明细 -> 计数递减 -> 删状态 key -> 发取消事件。

## 4. 批量状态查询
- 使用 Redis Pipeline 批量 `exists`
- miss 回源 DB 批量查询
- 回填命中项到 Redis

## 5. 一致性模型
- 主真相在 DB 明细表。
- Redis 是性能缓存，更新失败不回滚主事务。
- 异常路径依赖日志+指标+巡检修正。

## 6. 风险点
- 计数 `decrement` 没有下界保护，可能出现负数。
- Redis 与 DB 计数可能短暂偏差。

## 7. 监控建议
- `cache.update.failure` 指标（已有）
- 点赞/收藏成功率
- 计数偏差任务告警（DB 明细 count vs Redis count）

## 8. 排障 Runbook
1. 查 DB 是否存在互动明细（按 userId/postId）。
2. 查 Redis 状态 key 与 count key。
3. 对比事件是否已进 Outbox 并成功投递。

## 9. 改造建议
- 增加计数最小值保护（不小于 0）。
- 引入异步校准任务，定期修复 Redis 计数漂移。
- 对高热点文章可引入分片计数或批量合并写策略。
