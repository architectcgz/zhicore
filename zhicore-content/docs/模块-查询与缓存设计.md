# 模块设计：查询与缓存（接手版）

## 1. 模块目标
在保证正确性的前提下，降低文章详情与列表读取延迟，避免数据库热点，控制缓存击穿。

## 2. 关键类
- 查询实现：`PostQueryServiceImpl`
- 缓存装饰：`CacheAsidePostQuery`
- 应用入口：`PostApplicationService#getPostById/getPublishedPostsHybrid`

## 3. 数据装配职责
`PostQueryServiceImpl#getDetail` 会聚合三类数据：
- 元数据：PG `posts`
- 正文：Mongo（失败可降级）
- 统计：PG `post_stats`（可空，空则默认 0）

## 4. 缓存策略
### 4.1 详情缓存
- 正常结果：30 分钟 + 抖动
- 降级结果（内容拉取异常）：5 分钟
- 空值：1 分钟（防穿透）
- 回源时加分布式锁防击穿

### 4.2 列表缓存
- 作者列表、标签列表、最新列表：10 分钟 + 抖动
- 缓存 miss 回源后写入

## 5. 分页模型
- 支持 Offset + Cursor 混合分页
- 深分页建议 Cursor，减少大 offset 的数据库成本

## 6. 典型时序（详情）
1. 读 Redis
2. miss -> 尝试加锁
3. 回源读 PG + Mongo + stats
4. 组装 `PostDetailView`
5. 按结果类型写缓存

## 7. 降级语义
- Mongo 读取异常时，`contentDegraded=true`，内容可为空。
- 调用方可根据此标记决定前端提示或重试策略。

## 8. 风险点
- `GET /api/v1/posts` 的 `sort/status` 参数目前未实际使用。
- `PostVO.publishedAt` 目前取 `createdAt`，语义偏差。

## 9. 排障建议
1. 先确认缓存命中率和 key 是否存在。
2. 再看 Mongo 连通性与超时日志。
3. 若出现内容空但元数据有值，检查 `contentDegraded` 与 Mongo 文档是否缺失。

## 10. 改进建议
- 引入统一 Query DTO，避免 Controller 参数“声明了但没下传”。
- 将热门详情主动预热，减少冷启动抖动。
- 增加缓存一致性巡检（随机抽样对比缓存与数据库）。
