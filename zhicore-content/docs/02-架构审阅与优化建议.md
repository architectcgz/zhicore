- 方法：以代码实现为准，和现有 `docs/*` 设计文档交叉核对。
- 目标：输出“当前模块设计说明 + 已确认问题 + 可执行优化建议”。

## 2. 当前模块与功能设计（代码现状）

## 2.1 分层架构
- Interface 层：HTTP 接口与参数适配，核心入口为 `PostController`、`TagController`、`PostLikeController`、`PostFavoriteController`、`AdminPostController`。
- Application 层：用例编排与事务边界，核心为 `PostApplicationService`、`PostFacadeService`、`TagApplicationService`、命令处理器与工作流。
- Domain 层：`Post`、`Tag` 等聚合根，含状态机与领域事件（如 `PostCreatedDomainEvent`、`PostPublishedDomainEvent`）。
- Infrastructure 层：PG/Mongo 持久化、Redis 缓存与锁、RocketMQ + Outbox、调度任务、事件消费者。

## 2.2 Post 生命周期模块
- 创建草稿：`PostApplicationService#createPost` -> `CreateDraftWorkflow#execute`，先写 PG 元数据，再写 Mongo 内容，再发领域事件与 Outbox 集成事件。
- 更新文章：`PostApplicationService#updatePost` 混合使用“直接仓储写入 + 命令处理器（meta/content）”。
- 发布：`PostApplicationService#publishPost` -> `PublishPostWorkflow#execute`，校验作者、状态、内容存在后发布。
- 定时发布：`schedulePublish` 发送延迟事件，`PostScheduleConsumer` 回调 `executeScheduledPublish`。
- 删除与恢复：软删除由 `DeletePostHandler`，恢复由 `RestorePostHandler`。

## 2.3 查询与缓存模块
- 查询聚合：`PostQueryServiceImpl` 组合 PG 元数据 + Mongo 内容 + PG 统计。
- 缓存策略：`CacheAsidePostQuery` 实现详情/列表缓存、锁防击穿、降级短 TTL。

## 2.4 标签模块
- 标签查询：`TagApplicationService` 提供详情、列表、搜索、热门标签。
- 标签写入：`PostApplicationService#replacePostTags` 全量替换关系并发布 `PostTagsUpdatedIntegrationEvent`。

## 2.5 点赞/收藏模块
- 入口：`PostLikeApplicationService`、`PostFavoriteApplicationService`。
- 流程：先查 Redis 快速判重，再写库，再回写 Redis，最后发集成事件。

## 2.6 Outbox 与消息模块
- 写入侧：`OutboxEventPublisher` 事务内写 `outbox_event`。
- 投递侧：`OutboxEventDispatcher` 定时扫描 PENDING 并投递 RocketMQ，失败重试。
- 消费侧：定时发布、用户资料变更、统计更新等消费者独立处理。

## 2.7 草稿与作者信息模块
- 草稿：`DraftServiceImpl + CachedDraftService`（Mongo 持久化 + Redis 缓存）。
- 作者信息：创建时快照冗余，后续由 MQ 消费和定时任务回填。

## 2.8 调度与运维模块
- 调度：Outbox 投递、消费事件清理、不完整文章清理、作者信息回填。
- 监控：`OutboxMetrics` 采集堆积指标，Actuator 暴露指标。

## 3. 已确认问题与优化建议（按优先级）

## 3.1 P0（应优先修复）

| 问题 | 影响 | 代码证据 | 优化建议 |
|---|---|---|---|
| 定时发布可能提前执行（最大仅 2h 延迟，且消费时不校验 `scheduledAt`） | 用户设置“明天发布”可能在 2 小时后被提前发布 | `PostApplicationService.java:932-953`、`PostApplicationService.java:527-556` | 消费端增加 `now >= scheduledAt` 校验；未到时重投递/落任务表；长期建议改“任务表 + 调度器”精确触发 |
| 点赞/收藏仓储实现缺失，主链路不可落地 | 运行期 Bean 装配/持久化能力存在高风险，功能不可用 | `PostLikeRepository.java:14`、`PostFavoriteRepository.java:14`、`PostLikeApplicationService.java:39`、`PostFavoriteApplicationService.java:39` | 补齐 PG 实现（含唯一约束与幂等），补充对应 DDL 与集成测试 |
| Outbox 表要求 `aggregate_version NOT NULL`，但互动事件传 `null` | 互动事件写 Outbox 可能失败，导致接口报错或事件丢失 | `src/main/resources/db/schema.sql:93-95`、`PostLikeApplicationService.java:101-104`、`PostFavoriteApplicationService.java:100-103`、`PostLikeApplicationService.java:150-153`、`PostFavoriteApplicationService.java:147-150` | 明确 `aggregateVersion` 语义并保证非空（如使用文章版本或事件版本）；同步修正事件模型与校验 |
| `updateAuthorInfo` 用 `@Select` 执行 `UPDATE` | SQL 执行语义错误风险，作者信息同步不可靠 | `PostEntityMyBatisMapper.java:107-119` | 改为 `@Update`，并补“作者信息变更”回归测试 |
| 仅声明 `@Version` 但未启用乐观锁拦截器 | 并发写可能静默覆盖，版本字段保护失效 | `PostEntity.java:118-119`、`MyBatisPlusConfig.java:24-27` | 增加 `OptimisticLockerInnerInterceptor` 并在冲突时返回可识别业务错误 |

## 3.2 P1（建议近期修复）

| 问题 | 影响 | 代码证据 | 优化建议 |
|---|---|---|---|
| 列表接口 `sort/status` 参数未下传 | 接口契约与行为不一致 | `PostController.java:145-150`、`PostFacadeService.java:86-90` | 定义统一 Query DTO 并下传到仓储 |
| `PostStatus.valueOf(status)` 无保护 | 非法状态输入触发 500 风险 | `PostFacadeService.java:82-84` | 增加安全解析与参数错误码 |
| 详情 `publishedAt` 被错误映射为 `createdAt` | 展示语义错误 | `PostApplicationService.java:627`、`PostApplicationService.java:693`、`PostDetailView.java:23-100` | 在查询视图补充 `publishedAt` 并统一映射 |
| 光标分页解码忽略 `postId` | 同时间戳场景可能漏数/重复 | `PostApplicationService.java:889-903` | 光标使用 `(publishedAt, postId)` 组合游标并在 SQL 双字段比较 |
| 空值缓存标记无法区分“缓存空值”和“缓存未命中” | 防穿透机制失效 | `RedisCacheRepositoryImpl.java:54-57`、`CacheAsidePostQuery.java:87-93`、`CacheAsidePostQuery.java:123-127` | 在 `CacheRepository` 返回值中显式区分 `MISS/NULL/HIT` |
| 列表缓存 key 未包含 `size` | 不同分页大小会互相污染缓存 | `PostRedisKeys.java:72-73`、`CacheAsidePostQuery.java:148-166` | key 增加 `size`（必要时含排序条件） |
| `KEYS` 被用于缓存失效，且 `deletePattern` 默认关闭 | 大 key 空间下有阻塞风险，部分失效逻辑名存实亡 | `TagApplicationService.java:310-314`、`TagStatsEventHandler.java:219-222`、`RedisCacheRepositoryImpl.java:173-180`、`DeletePostHandler.java:87`、`PurgePostHandler.java:119` | 统一改为 `SCAN`，并把模式失效替换为版本化 key 或精确 key 删除 |
| Outbox 失败告警仍是 TODO | 失败事件不可观测，积压风险高 | `OutboxEventDispatcher.java:189-190` | 接入告警通道（企业IM/短信/Pager）并做演练 |
| 标注了 `@Async` 但未见 `@EnableAsync` | 事件处理可能仍同步执行，拉长请求尾延迟 | `PostApplication.java:15-20`、`TagStatsEventHandler.java:51/92/118`、`PostMongoDBSyncEventHandler.java:47/109` | 增加 `@EnableAsync` 与线程池配置，明确异步边界 |
| 标签域服务本地生成 ID 且批量创建吞异常 | ID 可靠性与数据完整性不足 | `TagDomainServiceImpl.java:125-127`、`TagDomainServiceImpl.java:156-162`、`TagDomainServiceImpl.java:214-216` | 接入统一 ID 服务；批量创建失败应 fail-fast 或返回失败明细并回滚 |
| `detachTag` 使用“读后全量替换” | 并发更新下有覆盖风险 | `PostFacadeService.java:129-134` | 改为差量更新（attach/detach）+ 版本校验 |
