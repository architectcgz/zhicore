# 模块设计：作者信息同步与补偿

## 1. 模块目标
作者信息模块负责在文章中维护作者快照（昵称、头像、资料版本），目标是：
- 查询文章时不依赖 user-service 实时联查。
- 用户资料更新后，文章冗余字段可最终收敛一致。
- 在 user-service 抖动时，通过补偿机制减少脏数据长期残留。

## 2. 数据模型与一致性基线
`posts` 表冗余字段：
- `owner_name`
- `owner_avatar_id`
- `owner_profile_version`

版本规则：
- 仅允许 `owner_profile_version` 向高版本更新（`< newVersion`）。
- 同版本或旧版本事件应被忽略，防止乱序覆盖。

## 3. 功能链路设计
### 3.1 创建文章时快照写入
1. `PostApplicationService.createPost` 调用 user-service 获取用户简档。
2. 成功则构造真实 `OwnerSnapshot`。
3. 失败则降级 `OwnerSnapshot.createDefault`（未知用户，version=0）。
4. 快照随文章主记录落库。

### 3.2 用户资料更新事件驱动同步
- 消费链路 A：`UserProfileUpdatedConsumer`（`user-profile-events`）
- 消费链路 B：`UserProfileChangedConsumer` + `PostAuthorSyncService`（`user-profile-changed`）
- 核心更新接口：`PostRepository.updateAuthorInfo(...)`

同步行为：
- 基于用户 ID 批量更新该用户文章的冗余字段。
- 通过版本号过滤旧消息。
- 局部场景会失效文章详情/作者列表缓存。

### 3.3 补偿与兜底
- 消费补偿事件：`AuthorInfoCompensationConsumer`
- 定时兜底：`AuthorInfoBackfillScheduler` 扫描“未知用户+默认版本”数据回填

## 4. 幂等与容错策略
- 数据幂等：`owner_profile_version` 版本过滤保证“高版本覆盖低版本”。
- 事件幂等：`UserProfileUpdatedConsumer` 使用 `consumed_events` 去重。
- 补偿幂等：`AuthorInfoCompensationConsumer` 通过 `post.hasDefaultAuthorInfo()` 判定是否需要更新。

## 5. 当前实现问题（需优化）
1. 创建链路目前走 `getOwnerSnapshot` 降级，但未直接触发补偿消息；即时补偿能力与设计注释不一致，主要依赖夜间 backfill。
2. `PostApplicationService` 内 `fillAuthorInfo/scheduleAuthorInfoCompensation` 为未接入路径，存在“设计在代码里、流程未启用”的维护风险。
3. `PostEntityMyBatisMapper.updateAuthorInfo` 使用 `@Select` 承载 `UPDATE` SQL，语义与框架惯例不匹配，影响可维护性与工具行为预期。
4. 补偿事件构造逻辑中存在 `aggregateVersion=null` 的路径，和 `outbox_event.aggregate_version NOT NULL` 约束冲突（若启用该路径将失败）。
5. 缓存失效依赖 `deletePattern`，但默认配置关闭该能力时只能记录日志，作者列表缓存可能滞后。
6. 同时维护两套用户资料事件消费链路（旧 topic 与新 topic），增加语义分叉与运维复杂度。

## 6. 优化建议
1. 明确唯一主链路：统一 `UserProfileUpdatedIntegrationEvent` + `consumed_events` 幂等，逐步下线旧 topic 消费。
2. 将创建降级后的补偿“显式启用”：创建时写补偿事件（含有效版本号），缩短未知作者暴露窗口。
3. 修正 Mapper 注解为 `@Update`，并补充集成测试验证版本过滤逻辑。
4. 缓存失效改为版本化 key 或精确 key 删除，避免生产依赖 pattern 删除。
5. 为作者同步增加监控：回填成功率、未知作者存量、平均修复时延（MTTR）。
