# 模块设计：管理端文章管理

## 1. 模块目标
管理端模块用于给运营/管理员提供内容治理能力。当前已落地能力：
- 条件分页查询文章。
- 管理员软删除文章。

## 2. 对外接口
- `GET /api/v1/admin/posts`：按关键词、状态、作者、分页查询。
- `DELETE /api/v1/admin/posts/{postId}`：管理员软删除。

## 3. 核心组件
- 接口层：`AdminPostController`
- 应用层：`AdminPostApplicationService`
- 仓储：`PostRepository.findByConditions/countByConditions/update`

## 4. 查询流程设计
1. 控制器接收筛选参数并传入应用层。
2. 应用层使用 `QueryParamValidator` 做参数规范化。
3. `status` 从枚举字符串转换为数据库状态码（`PostStatus.code`）。
4. 执行分页查询与总数查询。
5. 转换 `Post -> PostManageDTO`，返回 `PageResult`。

## 5. 删除流程设计
1. 根据 `postId` 查询文章，不存在直接报错。
2. 调用领域行为 `post.delete()` 切换为删除态。
3. 持久化更新。

说明：当前管理端删除是“软删除”，不包含正文物理清理。

## 6. 当前实现问题（需优化）
1. `authorName` 在 DTO 转换时被写死为空字符串，管理列表信息不完整。
2. 管理员权限校验未在该模块内显式体现（依赖外围网关/鉴权约定），审计可追踪性不足。
3. 删除接口无“删除原因/工单号/操作者上下文”字段，后续问责与回滚成本高。
4. 查询能力与 OpenAPI 描述存在扩展目标（审核、置顶、推荐）但代码尚未落地，文档与实现存在差距。

## 7. 优化建议
1. 补齐作者展示字段（直接使用 `posts.owner_name` 冗余快照）。
2. 增加管理员操作审计日志（操作者、目标资源、操作类型、原因、时间、请求ID）。
3. 删除操作扩展为“软删除 + 记录原因 + 可恢复窗口”标准流程。
4. 管理端分页结果增加更多治理字段（举报数、下架原因、最后审核时间）。
5. 为管理接口增加访问频控与操作告警，降低误操作放大风险。
