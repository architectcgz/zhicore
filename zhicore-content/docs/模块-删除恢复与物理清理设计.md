# 模块设计：删除、恢复与物理清理（接手版）

## 1. 模块目标
覆盖文章下线全链路：
- 软删除（业务下线）。
- 恢复（撤销下线）。
- 物理清理（彻底删除 PG/Mongo 数据）。

## 2. 入口接口
- `DELETE /api/v1/posts/{postId}`：作者软删除。
- `POST /api/v1/posts/{postId}/restore`：作者恢复。
- 物理清理：当前通过 `PurgePostHandler` 提供能力（未暴露稳定公开接口）。

## 3. 核心链路
- 软删除：`PostApplicationService#deletePost -> DeletePostHandler#handle`
- 恢复：`PostFacadeService#restorePost -> RestorePostHandler#handle`
- 物理清理：`PurgePostHandler#handle`

## 4. 数据与事件行为
- 软删除：更新 `posts.status=DELETED`，并发 `PostDeletedEvent`。
- 恢复：状态回到 `DRAFT`，并发恢复事件。
- 物理清理：删除 Mongo 内容 + 删除 PG 记录 + 缓存清理 + `PostPurgedEvent`。

## 5. 一致性策略
- 软删除/恢复：PG 内一致，缓存最终一致。
- 物理清理：Mongo + PG 跨存储顺序执行，失败时依赖重试或人工补偿。

## 6. 当前实现风险点
1. 删除正文内资源（图片/媒体）仅 TODO，可能产生对象存储垃圾。
2. `deletePattern` 在默认配置下关闭，部分“全量缓存失效”逻辑可能不生效。
3. 物理清理的管理员权限检查仍是 TODO，权限模型不完整。

## 7. 优化建议
1. 落地资源清理任务：解析正文 -> 异步删除 -> 失败重试。
2. 缓存失效从 `pattern` 转为精确 key 或版本化 key。
3. 补齐管理员权限与审计日志（操作者、原因、目标、时间）。
