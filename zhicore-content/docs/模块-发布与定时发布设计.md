# 模块设计：发布与定时发布（接手版）

## 1. 模块目标
该模块保证文章从草稿到公开发布的状态迁移正确、可追踪、可恢复，支持立即发布与定时发布两种路径。

## 2. 入口接口
- `POST /api/v1/posts/{postId}/publish`：立即发布
- `POST /api/v1/posts/{postId}/unpublish`：撤回发布
- `POST /api/v1/posts/{postId}/schedule`：设置定时发布
- `DELETE /api/v1/posts/{postId}/schedule`：取消定时发布

## 3. 核心类
- `PostApplicationService`
- `PublishPostWorkflow`
- 延迟执行：`PostScheduleConsumer`（消费端）+ `executeScheduledPublish`

## 4. 立即发布时序
1. Controller 获取 userId + postId。
2. `PostApplicationService#publishPost` 调 `PublishPostWorkflow#execute`。
3. Workflow 校验：
   - 操作人必须是作者
   - 文章不能是 DELETED
   - 内容必须存在（Mongo）
   - 写入状态不能是 `DRAFT_ONLY`
4. 聚合执行 `publish`，状态变更并 `postRepository.update`。
5. 领域事件发布 + 写 Outbox（`PostPublishedIntegrationEvent`）。

## 5. 定时发布时序
1. 设置 `scheduledAt` 并更新状态为 `SCHEDULED`。
2. 写两类集成事件到 Outbox：
   - 执行事件（含 delay level）
   - 审计/通知事件（Scheduled）
3. MQ 到期后消费者触发 `executeScheduledPublish(postId)`。
4. 重新检查状态：不是 `SCHEDULED` 则跳过（幂等）。
5. 满足条件则发布并再次发 `PostPublishedIntegrationEvent`。

## 6. 状态机（简化）
- `DRAFT -> PUBLISHED`
- `DRAFT/PUBLISHED -> SCHEDULED -> PUBLISHED`
- `PUBLISHED -> DRAFT`（撤回语义，具体由聚合规则定义）
- `* -> DELETED`（删除后不可发布）

## 7. 一致性与幂等
- 发布状态变更和 Outbox 写入在同事务。
- 定时执行端通过状态检查防重复发布。
- 下游应继续用 `eventId` 幂等消费。

## 8. 异常场景
- 内容缺失：拒绝发布。
- 非作者操作：拒绝。
- 定时消息迟到：状态已变化则安全跳过。

## 9. 监控与排障
- 监控项：
  - 发布成功率
  - 定时任务触发延迟（实际发布时间 - 计划发布时间）
  - 定时发布跳过次数
- 排障步骤：
  1. 查 `posts.status/scheduled_at/published_at`
  2. 查对应 Outbox 事件是否投递
  3. 查 MQ 消费日志

## 10. 改造建议
- 延迟级别目前离散，建议评估“精准定时任务表 + 调度器”方案。
- 建议补“定时发布幂等键”落库，进一步增强消费端鲁棒性。
