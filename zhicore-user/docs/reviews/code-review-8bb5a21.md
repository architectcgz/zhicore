## 新发现的问题

### 问题 1：拉黑操作获取三把锁，嵌套深度过深且与架构文档的锁获取顺序描述不完全一致

- 级别：**中**
- 涉及文件：`BlockApplicationService.java:56-100`
- 问题描述：当前实现获取 3 把锁（blockLock → followLock1 → followLock2），嵌套 4 层 try-finally，代码可读性较差。架构文档 06-可靠性与一致性设计 2.2 节描述的是"先获取 blockLock，再获取 followLock"（2 把锁），当前实现拆成了 3 把锁
- 功能正确性：3 把锁的方案在并发安全上是正确的——`followLock(min,max)` 和 `followLock(max,min)` 分别与 A→B 和 B→A 方向的正常关注操作互斥，覆盖了双向关注场景
- 修正建议（可选优化，不阻塞合并）：
  1. 考虑使用 Redisson 的 `RedissonMultiLock` 将 3 把锁合并为一次获取，减少嵌套层级
  2. 更新架构文档 2.2 节，补充拉黑操作实际需要 3 把锁的说明

---

## 做得好的地方

1. **关注锁键复用设计正确**：删除了 `blockFollowLock()` 方法，直接复用 `followLock()` 生成与正常关注操作完全一致的锁键，从根本上解决了锁键格式不匹配的问题
2. **锁获取顺序严格有序**：按 `blockLock → followLock(min→max) → followLock(max→min)` 固定顺序获取，所有拉黑操作遵循相同顺序，不会死锁
3. **登录降级处理得当**：Redis 异常时抛出 `INTERNAL_ERROR` 业务异常而非暴露原始 500，用户体验友好且日志中保留了完整异常栈便于排查
4. **Outbox 投递参数调优合理**：批量大小 100→50 + timeout=30s，在吞吐量和锁持有时间之间取得了合理平衡
