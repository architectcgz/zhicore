## 高级问题

### 问题 1：swap-and-flush 模式缺少原子性保证，存在数据丢失窗口

- 级别：**高**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：文档描述"刷写时使用 swap-and-flush 模式，新事件写入新 Map，不阻塞消费"，但没有说明 swap 操作的原子性如何保证。`ConcurrentHashMap` 本身不支持原子 swap——如果实现为"先 new 一个空 Map 赋值给引用，再取旧 Map 刷写"，在赋值与取旧 Map 之间存在竞态窗口：消费线程可能仍在向旧 Map 写入，而刷写线程已经开始遍历旧 Map。此外，如果刷写过程中（批量写 MongoDB + Redis）发生异常，旧 Map 中的数据已经从引用上摘除，但未成功持久化，数据将永久丢失。
- 影响范围：高并发场景下热度增量丢失或重复计算；刷写失败时缓冲数据不可恢复
- 修正建议：
  1. 明确 swap 操作的实现方式：建议使用 `AtomicReference<ConcurrentHashMap>.getAndSet(new Map)` 保证引用替换的原子性
  2. 刷写失败时必须有补偿策略：将失败的批次重新放回缓冲区，或写入本地磁盘 WAL（Write-Ahead Log）
  3. 在文档中补充 swap-and-flush 的伪代码或时序图，明确线程安全边界

### 问题 2：Redis 总榜 Sorted Set 无限膨胀，缺少淘汰机制

- 级别：**高**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：`ranking:posts:hot` 总榜使用 ZINCRBY 增量更新，但文档没有说明任何成员淘汰机制。随着文章数量持续增长，Sorted Set 中的成员数将无限膨胀。即使文章已经因时间衰减分数趋近于零，仍然占据 Redis 内存。日/周/月榜有 TTL 自动过期，但总榜 TTL 仅 5 分钟——这与 ZINCRBY 增量更新的描述矛盾：如果总榜每 5 分钟过期，增量更新毫无意义，每次都需要全量重建；如果不过期，则面临无限膨胀。
- 影响范围：Redis 内存持续增长，最终触发 maxmemory 淘汰策略，可能导致其他缓存被驱逐；大 Sorted Set 的 ZREVRANGE 性能也会退化
- 修正建议：
  1. 明确总榜的更新策略：是增量更新（ZINCRBY）还是快照全量刷新，二者只能选一，当前文档自相矛盾
  2. 如果采用增量更新，必须定期执行 ZREMRANGEBYSCORE 清理低分成员，或 ZREMRANGEBYRANK 只保留 Top N（如 Top 10000）
  3. 补充 Redis 内存容量预估：按文章增长速率估算 Sorted Set 大小上限

### 问题 3：取消互动（取消点赞/取消收藏）的热度衰减不对称，缺少负向增量处理

- 级别：**高**
- 涉及文档：`02-核心功能设计.md`、`01-架构总览与设计决策.md`
- 问题描述：热度公式 `raw_score = view * 1 + like * 5 + comment * 10 + favorite * 8` 只定义了正向增量。文档中互动事件来源为"浏览/点赞/评论/收藏"，但完全没有提及取消点赞、取消收藏、删除评论等负向操作。如果用户取消点赞，热度分数不会递减，导致热度只增不减（除了时间衰减）。恶意用户可以反复点赞/取消点赞来刷高热度。
- 影响范围：排行榜热度失真，存在刷榜漏洞；用户取消互动后排行不反映真实热度
- 修正建议：
  1. 明确负向事件的处理策略：取消点赞时 ZINCRBY -5，取消收藏时 ZINCRBY -8，删除评论时 ZINCRBY -10
  2. 在本地缓冲区的 `ScoreIncrement` 中支持负值累加
  3. 增加防刷机制：同一用户对同一文章的互动在时间窗口内去重（如 1 小时内重复点赞/取消只计一次）
  4. 在文档中明确列出所有支持的事件类型，包括正向和负向

### 问题 4：缓冲区刷写与 Redis/MongoDB 双写缺少一致性保证

- 级别：**高**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：刷写操作描述为"批量写入 MongoDB + 更新 Redis Sorted Set"，但没有说明两者的写入顺序和失败处理。如果 MongoDB 写入成功但 Redis 更新失败（网络抖动、Redis 主从切换），排行缓存与持久化数据将不一致——用户查询到的排行（来自 Redis）与实际数据（MongoDB）产生偏差。反之亦然。由于缓冲区数据在 swap 后已丢弃，无法重试。
- 影响范围：排行查询结果与实际热度数据不一致；部分文章排名异常偏高或偏低
- 修正建议：
  1. 明确写入顺序：建议先写 MongoDB（权威数据源），成功后再更新 Redis
  2. Redis 更新失败时的补偿策略：记录失败的 batch，由定时任务从 MongoDB 重建 Redis 缓存
  3. 考虑引入"快照生成时全量刷新 Redis"作为兜底一致性机制，而非仅依赖增量更新

### 问题 5：阈值触发刷写（1000 条）与定时刷写（5s）的竞态条件未说明

- 级别：**高**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：缓冲区有两个刷写触发条件——"每 5 秒定时刷写"和"累计 1000 条记录"。文档没有说明两者的协调机制。如果定时器线程和阈值触发线程同时检测到需要刷写，可能出现：(1) 两个线程同时执行 swap，导致其中一个拿到空 Map 做无效刷写；(2) 两个线程先后 swap，中间短暂窗口的事件被第二次 swap 吞掉。此外，"1000 条记录"的计数方式不明确——是 Map 的 key 数量（去重后的文章数）还是累计事件条数？两者差异巨大。
- 影响范围：并发刷写导致数据丢失或重复写入；阈值判断不准确导致刷写时机偏移
- 修正建议：
  1. 明确只有一个线程负责执行 swap-and-flush，阈值触发只是"提前唤醒"定时刷写线程，而非独立执行刷写
  2. 使用 ReentrantLock 或 synchronized 保证同一时刻只有一个刷写操作在执行
  3. 明确"1000 条"的语义：建议使用 AtomicLong 计数器记录累计事件数（非去重），阈值触发后重置

### 问题 6：互动事件消费缺少幂等性设计

- 级别：**高**
- 涉及文档：`01-架构总览与设计决策.md`、`02-核心功能设计.md`
- 问题描述：RocketMQ 消费者在网络抖动、Broker 重平衡或消费者重启时可能重复投递消息。文档没有提及任何消费幂等机制。同一条"点赞"事件被重复消费时，热度分数会被多次累加（+5 变成 +10、+15...），直接导致排行榜数据失真。由于本地缓冲区按 PostId 聚合增量，重复事件会被累加到同一个 ScoreIncrement 中，无法在刷写阶段去重。
- 影响范围：排行榜热度数据不准确，重复消费越多偏差越大；极端情况下某篇文章因重复消费被推到榜首
- 修正建议：
  1. 在消费端引入消息去重机制：基于 `messageId` 或业务唯一键（如 `userId + postId + eventType + timestamp`）做去重
  2. 去重窗口建议覆盖缓冲区刷写周期的 2-3 倍（如 15s），可使用 Redis SET + TTL 或本地 BloomFilter
  3. 在文档中明确 RocketMQ 消费模式（集群消费 / 广播消费）和重试策略

## 中级问题

### 问题 7：优雅停机刷写缓冲区的可靠性未充分设计

- 级别：**中**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：文档提到"shutdown hook 中强制刷写缓冲区残留数据"，但 shutdown hook 的执行时间有限（JVM 默认无超时，但 K8s terminationGracePeriodSeconds 通常 30s）。如果缓冲区积压量大，或 MongoDB/Redis 响应慢，刷写可能在 hook 超时前未完成。此外，SIGKILL 信号会直接终止进程，shutdown hook 不会执行。
- 影响范围：服务重启或 Pod 被强制终止时缓冲区数据丢失
- 修正建议：
  1. 明确 shutdown hook 的超时策略和优先级
  2. 考虑在缓冲区 swap 后将待刷写数据先写入本地磁盘（WAL），启动时检查并恢复未完成的刷写
  3. 在文档中评估最坏情况下的数据丢失量（5s 窗口 * 峰值 QPS）

### 问题 8：时间衰减仅在快照生成时计算，排行实时性与准确性存在矛盾

- 级别：**中**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：文档明确写到"衰减因子在每次快照生成时重新计算，非实时衰减"。但 Redis Sorted Set 中的分数是通过 ZINCRBY 增量更新的 raw_score，并未应用衰减因子。这意味着两次快照生成之间（总榜 5 分钟，日/周/月榜 1 小时），Redis 中的排行完全基于未衰减的原始分数。一篇 30 天前的高互动文章在 Redis 中的分数可能远高于一篇 1 天前的中等互动文章，但快照中（应用衰减后）排名恰好相反。用户在不同时间点查询会看到跳跃式的排名变化。
- 影响范围：快照刷新前后排行结果可能剧烈变化，用户体验不一致
- 修正建议：
  1. 明确 Redis 缓存中存储的是 final_score（含衰减）还是 raw_score（不含衰减），当前文档描述矛盾
  2. 如果 Redis 存 raw_score，查询时需在应用层实时计算衰减并重排序，但这会失去 Sorted Set 的排序优势
  3. 建议 Redis 中直接存储 final_score，每次快照生成时全量刷新 Redis（而非增量 ZINCRBY）

### 问题 9：创作者排行和话题排行的聚合逻辑完全缺失

- 级别：**中**
- 涉及文档：`02-核心功能设计.md`、`01-架构总览与设计决策.md`
- 问题描述：服务定位明确包含"创作者影响力排名"和"话题热度排名"，API 也定义了 `/creators/hot` 和 `/topics/hot` 接口，但整个核心功能设计文档只描述了文章热度的计算流程。创作者排行如何从文章热度聚合？是所有文章分数求和，还是取 Top N 文章？话题排行如何关联文章？这些聚合逻辑完全没有设计。
- 影响范围：实现时缺少设计依据，可能导致各开发者理解不一致；聚合查询的性能特征未评估
- 修正建议：
  1. 补充创作者排行的聚合公式：建议为"该创作者所有文章 final_score 之和"或"Top 10 文章 final_score 之和"
  2. 补充话题排行的聚合公式：需明确文章与话题的关联方式（标签？分类？）以及聚合维度
  3. 说明聚合计算的时机：是在快照生成时从文章排行数据二次聚合，还是独立维护增量

### 问题 10：MongoDB 排行快照文档使用嵌套数组，存在 16MB 文档大小限制风险

- 级别：**中**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：排行快照将所有排行项存储在单个文档的 `items` 数组中。MongoDB 单文档上限 16MB。假设每个 item 约 200 字节（targetId + score + rank + metadata），一个文档最多存储约 8 万条记录。如果总榜文章数超过此限制，文档写入将失败。即使未达上限，大文档的读写性能也会显著下降（全量读取、全量替换）。
- 影响范围：文章数量增长后快照写入失败；大文档读写延迟增加
- 修正建议：
  1. 快照只存储 Top N（如 Top 1000），而非全量排行数据
  2. 如需存储更多数据，改为每个排行项一个文档，通过 `type + period + rank` 复合索引查询
  3. 在文档中明确快照存储的数据量上限

### 问题 11：缓存 miss 回源 MongoDB 时缺少防击穿/防雪崩机制

- 级别：**中**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：非功能设计中提到"查询优先读 Redis，miss 时回源 MongoDB 并回填缓存"，但没有说明缓存击穿和缓存雪崩的防护措施。总榜 TTL 仅 5 分钟，过期瞬间大量并发请求同时 miss，全部穿透到 MongoDB，可能导致 MongoDB 瞬时负载飙升。此外，如果某个排行类型在 MongoDB 中也不存在（如新上线的排行维度），会产生缓存穿透。
- 影响范围：缓存过期瞬间 MongoDB 负载激增；缓存穿透导致无效查询持续打到 DB
- 修正建议：
  1. 引入互斥锁（如 Redisson 分布式锁）防止缓存击穿：miss 时只允许一个请求回源，其余等待
  2. 对空结果缓存短 TTL（如 30s）防止缓存穿透
  3. 考虑使用逻辑过期策略：缓存不设物理 TTL，由后台线程异步刷新

### 问题 12：快照生成定时任务缺少分布式锁和失败重试机制

- 级别：**中**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：文档提到"定时任务每 5 分钟生成总榜快照，日/周/月榜每小时生成一次"，但没有说明多实例部署时如何避免重复执行。如果 ranking 服务部署多个实例，每个实例都会触发定时任务，导致同一时间段生成多份快照，浪费计算资源且可能产生数据冲突（多个实例同时 upsert 同一快照文档）。此外，快照生成失败时的重试策略也未说明。
- 影响范围：多实例重复计算浪费资源；并发 upsert 可能导致快照数据不完整
- 修正建议：
  1. 使用分布式锁（如 Redisson 分布式锁）确保同一时刻只有一个实例执行快照生成
  2. 快照生成失败时记录失败状态，下一个周期优先重试失败的快照
  3. 在文档中明确定时任务的调度框架（Spring @Scheduled / XXL-Job / 其他）

### 问题 13：排行查询 API 缺少文章状态过滤说明

- 级别：**中**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：排行榜中可能包含已删除、已下架或被封禁的文章。文档没有说明排行数据中如何处理这些异常状态的文章。如果文章在 content-service 中被删除，但 ranking 服务的 Redis Sorted Set 和 MongoDB 快照中仍保留该文章的排行数据，用户查询排行榜时会看到已不存在的文章。
- 影响范围：排行榜展示已删除/下架文章，用户点击后 404；排行位次被无效数据占据
- 修正建议：
  1. 监听文章状态变更事件（删除/下架/封禁），从 Redis Sorted Set 中 ZREM 对应成员
  2. 快照生成时过滤异常状态文章，或在查询时做二次过滤
  3. 在文档中明确跨服务数据一致性的处理策略

### 问题 14：浏览事件权重为 1 但量级远大于其他互动，存在刷榜风险

- 级别：**中**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：浏览事件权重虽然最低（1），但浏览量通常是点赞量的 50-100 倍。一篇文章 10000 次浏览贡献 10000 分，而 200 次点赞仅贡献 1000 分。浏览事件极易被刷量（爬虫、脚本刷新），且文档没有提及任何浏览去重或频率限制机制（如同一用户同一文章 N 分钟内只计一次浏览）。
- 影响范围：浏览刷量可轻易操纵排行榜；热度分数被浏览量主导，点赞/评论/收藏的区分度被稀释
- 修正建议：
  1. 引入浏览去重机制：同一用户对同一文章在时间窗口内（如 30 分钟）只计一次有效浏览
  2. 考虑对浏览分数设置上限（如单篇文章浏览贡献最多 5000 分），避免浏览量完全主导热度
  3. 在文档中补充防刷策略的设计说明

### 问题 15：排行快照 metadata 中冗余存储文章标题/作者名，缺少数据同步机制

- 级别：**中**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：MongoDB 快照文档的 `metadata` 字段存储了 `title` 和 `authorName`，这些数据的权威源在 content-service。如果文章标题被修改或作者改名，ranking 服务中的快照数据不会同步更新，导致排行榜展示过期信息。文档没有说明 metadata 的更新机制。
- 影响范围：排行榜展示的文章标题/作者名与实际不一致
- 修正建议：
  1. 方案 A：快照中不存储 metadata，查询时通过 RPC 或缓存从 content-service 获取最新信息
  2. 方案 B：监听文章/用户信息变更事件，异步更新快照中的 metadata
  3. 在文档中明确 metadata 的一致性要求和更新策略

## 低级问题

### 问题 16：API 响应中 rank 字段语义不明确

- 级别：**低**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：响应示例中包含 `rank` 字段（值为 1），但没有说明 rank 是在快照生成时预计算的，还是查询时根据分页位置动态计算的。如果是预计算，分页查询第 2 页的第一条 rank 应该是 21（假设 size=20），但如果是动态计算则需要额外逻辑。此外，rank 在并发更新场景下可能不连续（如中间有文章被删除）。
- 修正建议：明确 rank 的计算方式和语义，说明是否保证连续性

### 问题 17：架构文档缺少错误码定义

- 级别：**低**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：API 接口设计只给出了 200 成功响应示例，没有定义任何错误码。排行类型不存在、分页参数非法、服务降级等场景的错误响应格式和错误码完全缺失。
- 修正建议：补充错误码表，至少覆盖参数校验失败、排行类型不存在、服务限流、内部错误等场景

### 问题 18：可观测性指标定义不够具体

- 级别：**低**
- 涉及文档：`02-核心功能设计.md`
- 问题描述：非功能设计中提到"缓冲区大小指标、刷写延迟直方图、缓存命中率、快照生成耗时"，但没有给出具体的 metric 名称、标签维度和告警阈值。例如缓冲区大小超过多少应该告警？刷写延迟 P99 超过多少算异常？缓存命中率低于多少需要介入？
- 修正建议：
  1. 定义具体的 metric 名称（如 `ranking_buffer_size`、`ranking_flush_duration_seconds`）
  2. 明确关键指标的告警阈值
  3. 补充链路追踪（traceId）在事件消费和刷写流程中的传播方式

### 问题 19：ADR 中"5s 窗口内的数据丢失风险可接受"缺少量化评估

- 级别：**低**
- 涉及文档：`01-架构总览与设计决策.md`
- 问题描述：ADR-1 声明"5s 窗口内的数据丢失风险可接受（排行榜非强一致场景）"，但没有量化分析。在峰值 QPS 下，5s 窗口可能积累数千条互动事件，丢失后对排行结果的影响程度未评估。"可接受"应该有数据支撑，而非主观判断。
- 修正建议：补充量化分析——按预估峰值 QPS 计算 5s 窗口最大丢失事件数，评估对排行 Top 100 的影响幅度
