# 架构总览与设计决策

## 1. 服务定位

zhicore-ranking 是 zhicore 微服务体系中的排行榜服务，负责文章热度、创作者影响力、话题热度等多维度排行的计算与查询。

### 1.1 核心职责

- 文章热度排行：总榜、日榜、周榜、月榜
- 创作者排行：基于作品综合热度的创作者影响力排名
- 话题排行：基于关联文章热度的话题排名
- 热度计算：基于加权公式 + 时间衰减的热度评分引擎

### 1.2 非职责

| 能力 | 负责方 |
|------|--------|
| 文章内容管理 | content-service |
| 用户互动（点赞/收藏/评论） | content-service / comment-service |
| 推荐算法与个性化推荐 | recommendation-service（规划中） |

### 1.3 上下游依赖

```
              ┌─────────────┐
              │   Gateway   │
              └──────┬──────┘
                     │ HTTP
              ┌──────▼──────┐
              │   ranking   │
              │   (8108)    │
              └──┬───┬───┬──┘
         ┌───────┤   │   ├───────┐
         │       │   │   │       │
   ┌─────▼──┐ ┌─▼───▼─┐ ┌──────▼──────┐
   │ MongoDB │ │ Redis │ │  RocketMQ   │
   │(排行数据)│ │(缓存) │ │(互动事件消费)│
   └────────┘ └───────┘ └─────────────┘
```

- 输入：网关转发的排行查询请求、RocketMQ 互动事件（浏览/点赞/评论/收藏/取消点赞/取消收藏/删除评论）、文章状态变更事件（删除/下架/封禁/恢复）
- 输出：MongoDB 排行数据持久化、Redis 缓存
- 消费保证：RocketMQ 集群消费模式，消费端基于业务唯一键去重，确保幂等性（详见核心功能设计 2.2 节）

## 2. 技术选型

| 技术 | 用途 | 选型理由 |
|------|------|---------|
| Java 17 | 运行时 | LTS 版本，稳定可靠 |
| Spring Boot 3.2.4 | 应用框架 | 生态成熟 |
| Spring Cloud Alibaba | 微服务治理 | Nacos 注册/配置，Sentinel 限流 |

| Knife4j + OpenAPI 3 | API 文档 | 自动生成接口文档 |

## 3. 分层架构

采用标准三层架构，排行榜服务以事件消费、热度聚合和排行查询为核心。

```
┌──────────────────────────────────────────────────┐
│  Controller 层（controller/）                     │
│  排行查询参数校验、分页处理、统一响应封装            │
├──────────────────────────────────────────────────┤
│  Service 层（service/）                           │
│  inbox 写入、热度聚合、快照刷新、查询编排           │
├──────────────────────────────────────────────────┤
│  Infrastructure 层                                │
│  MongoDB 持久化、Redis 缓存、MQ 消费者、定时调度     │
└──────────────────────────────────────────────────┘
```

- Controller → Service：参数接收与校验，委托 Service 查询排行数据
- Service → Infrastructure：读写 MongoDB 排行快照、Redis 缓存
- MQ Consumer → Service：消费互动事件，先写 durable inbox，再进入异步聚合

## 4. 关键架构决策记录（ADR）

### ADR-1：消费后先写 Durable Inbox，再异步聚合

- 背景：旧方案将 MQ 事件先放 JVM 内存，再按 5 秒批量刷写；实例宕机时窗口内事件会直接丢失
- 决策：MQ consumer 只负责把事件持久化到 MongoDB `ranking_event_inbox`，随后由定时聚合作业更新权威热度状态
- 理由：
  - 将“消息到达 ranking”与“热度已完成聚合”拆成两个可恢复阶段
  - ACK 前先完成业务侧持久化，避免 JVM 内存窗口丢数
  - 便于做失败重试、超时接管和消费侧幂等
- 代价：链路从单阶段变成 inbox + 聚合 + 快照三阶段，实现复杂度更高
- 状态：已采纳

### ADR-2：MongoDB 同时承担权威热度状态与历史归档

- 背景：热度公式包含时间衰减，若直接增量更新 Redis `final_score`，会把未衰减的新分与已衰减旧分混算
- 决策：MongoDB 存储 `ranking_post_hot_state` 权威状态，Redis 只存当前快照；历史榜单继续归档到 MongoDB
- 理由：
  - 权威层保存净计数，可在权重或衰减规则变更后重算
  - Redis 严格只承载某一时刻计算出的 `final_score`
  - MongoDB 文档结构适合保存文章元数据、作者、话题和历史快照
- 代价：查询链路需要区分“当前排行缓存”和“历史归档”
- 状态：已采纳

### ADR-3：Redis Sorted Set 作为当前排行快照缓存

- 背景：排行榜查询是高频读场景，需要低延迟响应
- 决策：将当前总榜、日榜、周榜、月榜的 Top N 快照写入 Redis Sorted Set
- 理由：
  - Sorted Set 的 ZREVRANGE 天然适合按分数取 Top N
  - 通过全量替换而不是增量 `ZINCRBY`，可保持快照分数语义稳定
  - 不同时间维度使用不同 Key，和归档快照边界清晰
- 代价：快照刷新时会有批量写入开销，需限制 Top N
- 状态：已采纳

### ADR-4：时间衰减半衰期 7 天

- 背景：热度排行需要体现"时效性"，老文章不应长期霸榜
- 决策：热度公式引入指数衰减因子，半衰期设为 7 天
- 理由：
  - 7 天半衰期平衡了"新内容曝光"与"优质内容沉淀"
  - 指数衰减比线性衰减更符合内容热度的自然消退规律
  - 半衰期可通过配置调整，无需改代码
- 状态：已采纳
