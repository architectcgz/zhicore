# 架构总览与设计决策

## 1. 服务定位

zhicore-ranking 是 zhicore 微服务体系中的排行榜服务，负责文章热度、创作者影响力、话题热度等多维度排行的计算与查询。

### 1.1 核心职责

- 文章热度排行：总榜、日榜、周榜、月榜
- 创作者排行：基于作品综合热度的创作者影响力排名
- 话题排行：基于关联文章热度的话题排名
- 热度计算：基于加权公式 + 时间衰减的热度评分引擎

### 1.2 非职责

| 能力 | 负责方 |
|------|--------|
| 文章内容管理 | content-service |
| 用户互动（点赞/收藏/评论） | content-service / comment-service |
| 推荐算法与个性化推荐 | recommendation-service（规划中） |

### 1.3 上下游依赖

```
              ┌─────────────┐
              │   Gateway   │
              └──────┬──────┘
                     │ HTTP
              ┌──────▼──────┐
              │   ranking   │
              │   (8108)    │
              └──┬───┬───┬──┘
         ┌───────┤   │   ├───────┐
         │       │   │   │       │
   ┌─────▼──┐ ┌─▼───▼─┐ ┌──────▼──────┐
   │ MongoDB │ │ Redis │ │  RocketMQ   │
   │(排行数据)│ │(缓存) │ │(互动事件消费)│
   └────────┘ └───────┘ └─────────────┘
```

- 输入：网关转发的排行查询请求、RocketMQ 互动事件（浏览/点赞/评论/收藏/取消点赞/取消收藏/删除评论）、文章状态变更事件（删除/下架/封禁/恢复）
- 输出：MongoDB 排行数据持久化、Redis 缓存
- 消费保证：RocketMQ 集群消费模式，消费端基于业务唯一键去重，确保幂等性（详见核心功能设计 2.2 节）

## 2. 技术选型

| 技术 | 用途 | 选型理由 |
|------|------|---------|
| Java 17 | 运行时 | LTS 版本，稳定可靠 |
| Spring Boot 3.2.4 | 应用框架 | 生态成熟 |
| Spring Cloud Alibaba | 微服务治理 | Nacos 注册/配置，Sentinel 限流 |
| MongoDB | 排行数据持久化 | Schema 灵活，适合排行快照的文档结构 |

| RocketMQ | 消息队列 | 消费互动事件驱动热度更新 |
| MapStruct | 对象映射 | 编译期生成，零运行时开销 |
| Knife4j + OpenAPI 3 | API 文档 | 自动生成接口文档 |

## 3. 分层架构

采用标准三层架构（无 DDD），排行榜服务以计算和缓存为核心，无需复杂领域建模。

```
┌──────────────────────────────────────────────────┐
│  Controller 层（controller/）                     │
│  排行查询参数校验、分页处理、统一响应封装            │
├──────────────────────────────────────────────────┤
│  Service 层（service/）                           │
│  热度计算引擎、排行聚合逻辑、本地缓冲区管理         │
├──────────────────────────────────────────────────┤
│  Infrastructure 层                                │
│  MongoDB 持久化、Redis 缓存、MQ 消费者、本地缓冲刷写│
└──────────────────────────────────────────────────┘
```

- Controller → Service：参数接收与校验，委托 Service 查询排行数据
- Service → Infrastructure：读写 MongoDB 排行快照、Redis 缓存
- MQ Consumer → Service：消费互动事件，触发热度增量更新

## 4. 关键架构决策记录（ADR）

### ADR-1：本地聚合缓冲 + 批量刷写

- 背景：互动事件（浏览/点赞等）QPS 高，每条事件都直接写 MongoDB/Redis 会产生大量小写入
- 决策：在内存中维护本地聚合缓冲区，按 5s 间隔或 1000 条阈值批量刷写
- 理由：
  - 将高频小写入合并为低频批量写入，降低存储层压力
  - 同一文章在缓冲窗口内的多次互动可合并为一次热度增量
  - 5s 窗口内的数据丢失风险可接受（排行榜非强一致场景）
- 代价：服务重启时缓冲区未刷写数据会丢失，排行数据存在秒级延迟
- **量化评估**：按预估峰值 2000 QPS 计算，5s 窗口最多积累 10000 条互动事件。假设这些事件均匀分布在 500 篇文章上，每篇文章平均丢失 20 条事件（约 20-200 分增量）。对于 Top 100 文章（典型 final_score > 10000），单次丢失对排名的影响 < 2%。结合 WAL 兜底机制（见核心功能设计 2.3 节），正常关闭场景下数据丢失量趋近于零，仅 SIGKILL 强制终止时存在上述最坏情况
- 状态：已采纳

### ADR-2：MongoDB 存储排行快照而非实时计算

- 背景：排行榜可实时聚合计算，也可定期生成快照
- 决策：定时任务生成排行快照存入 MongoDB，查询时直接读取快照
- 理由：
  - 排行榜数据变化频率远低于查询频率，快照模式读性能更优
  - MongoDB 文档结构天然适合存储排行列表（嵌套数组）
  - 快照可保留历史版本，支持"昨日排行"等回溯查询
- 代价：排行数据非实时，更新周期取决于快照生成频率
- 状态：已采纳

### ADR-3：Redis Sorted Set 作为热排行缓存

- 背景：排行榜查询是高频读场景，需要低延迟响应
- 决策：将 Top N 排行数据缓存到 Redis Sorted Set
- 理由：
  - Sorted Set 的 ZREVRANGE 天然支持按分数降序取 Top N，O(logN+M) 复杂度
  - 增量更新可通过 ZINCRBY 原子操作完成，无需全量刷新
  - 不同时间维度（日/周/月）使用不同 Key，互不干扰
- 代价：Redis 内存占用随排行维度增加线性增长，需监控
- 状态：已采纳

### ADR-4：时间衰减半衰期 7 天

- 背景：热度排行需要体现"时效性"，老文章不应长期霸榜
- 决策：热度公式引入指数衰减因子，半衰期设为 7 天
- 理由：
  - 7 天半衰期平衡了"新内容曝光"与"优质内容沉淀"
  - 指数衰减比线性衰减更符合内容热度的自然消退规律
  - 半衰期可通过配置调整，无需改代码
- 状态：已采纳
