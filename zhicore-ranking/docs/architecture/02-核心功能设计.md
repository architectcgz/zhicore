# 核心功能设计

## 1. 热度计算引擎

### 1.1 热度权重公式

```
raw_score = view * 1 + like * 5 + comment * 10 + favorite * 8
```

| 互动类型 | 权重 | 设计理由 |
|---------|------|---------|
| view（浏览） | 1 | 最轻量互动，量大但价值密度低 |

#### 浏览事件防刷策略

- **用户级去重**：同一用户对同一文章在 30 分钟内只计一次有效浏览，去重基于 Redis `SET ranking:view:dedup:{userId}:{postId}` + 30min TTL
- **单篇浏览分数上限**：单篇文章的浏览贡献分数上限为 5000 分（即 5000 次有效浏览），超出后浏览事件不再累加热度。当前上限值由 Redis `ranking:view:cap:{postId}` 维护
- **匿名用户处理**：未登录用户按 IP + UserAgent 指纹去重，时间窗口同为 30 分钟

| like（点赞） | 5 | 主动正向反馈，价值中等 |
| favorite（收藏） | 8 | 深度认可，用户愿意后续回看 |
| comment（评论） | 10 | 最高价值互动，代表深度参与 |
| unlike（取消点赞） | -5 | 撤销正向反馈，热度相应递减 |
| unfavorite（取消收藏） | -8 | 撤销收藏行为 |
| uncomment（删除评论） | -10 | 撤销评论互动 |

#### 负向事件处理规则

- 取消点赞、取消收藏、删除评论以 `countDelta < 0` 的形式进入 inbox
- 聚合到 `ranking_post_hot_state` 时，各维度净计数允许递减，但最小值被钳制为 `0`
- 当前消费侧的幂等主键是 `eventId`，避免重复应用相同负向事件

### 1.2 时间衰减公式

```
decay_factor = 0.5 ^ (hours_since_publish / (7 * 24))
final_score  = raw_score * decay_factor
```

- 半衰期：7 天（168 小时），即发布 7 天后热度衰减为原始值的 50%
- 14 天后衰减为 25%，21 天后衰减为 12.5%
- 衰减因子在每次快照生成时重新计算，非实时衰减
- **Redis 中存储 final_score**：Redis Sorted Set 中的分数为含衰减的 final_score，而非 raw_score。每次快照生成时，从 MongoDB 读取所有文章的 raw_score 和 publishTime，重新计算 final_score 后全量刷新 Redis。两次快照之间的增量事件仅更新 MongoDB 中的 raw_score，不直接 ZINCRBY Redis（避免未衰减分数与已衰减分数混合累加）
- **增量事件的 Redis 更新策略**：聚合作业只更新 MongoDB 权威状态，Redis 排行数据完全由快照任务全量刷新。这保证了 Redis 中的排行始终基于统一时间点计算的 final_score，避免排名跳跃

## 2. 数据流与 Durable Inbox

### 2.1 互动事件处理流程

```
RocketMQ      Consumer           Inbox              Aggregation            Snapshot            Redis
  │ 互动事件     │                  │                    │                    │                 │
  │────────────>│ 反序列化/校验      │                    │                    │                 │
  │             │─────────────────>│ saveEvent()        │                    │                 │
  │             │                  │ ranking_event_inbox │                    │                 │
  │             │                  │                    │ claim + merge      │                 │
  │             │                  │───────────────────>│ ranking_post_hot_state
  │             │                  │                    │                    │ rebuild topN    │
  │             │                  │                    │                    │────────────────>│
```

### 2.2 消费幂等性设计

- **消费模式**：RocketMQ 集群消费模式（CLUSTERING），同一 ConsumerGroup 内消息只被一个实例消费
- **事件唯一性**：`ranking_event_inbox.eventId` 作为主键，同一事件重复投递时只会保留一条
- **双层幂等**：
  - 第一层：写 inbox 时依赖 `eventId` 唯一键拦截重复消费
  - 第二层：聚合到 `ranking_post_hot_state` 时，使用 `recentAppliedEventIds` 防止“状态已更新但 inbox 尚未置 DONE”导致的重复应用
- **重试策略**：聚合作业失败时将 inbox 记录重新置回 `NEW` 并增加 `retryCount`；超过 `maxRetry` 后置为 `FAILED`
- **租约接管**：被 claim 为 `PROCESSING` 的记录带 `leaseUntil`，若实例超时未完成，其他实例可重新接管

### 2.3 Durable Inbox 与权威热度状态设计

- `ranking_event_inbox` 记录消费到的单条事件，核心字段：
  - `eventId`
  - `postId`
  - `authorId`
  - `metricType`
  - `countDelta`
  - `occurredAt`
  - `publishedAt`
  - `status`
  - `leaseUntil`
  - `retryCount`
- `ranking_post_hot_state` 记录文章级权威热度状态，核心字段：
  - `postId`
  - `authorId`
  - `topicIds`
  - `publishedAt`
  - `viewCount`
  - `likeCount`
  - `favoriteCount`
  - `commentCount`
  - `rawScoreCache`
  - `recentAppliedEventIds`
- 聚合作业按 `postId` 合并一批 inbox 事件，写入一次 `ranking_post_hot_state`
- `rawScoreCache` 是当前权重下的缓存值，不是唯一权威值；真正可重算的权威信息仍然是各互动净计数

## 3. API 接口设计

### 3.1 接口总览

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/ranking/posts/hot` | 文章热度总榜 |
| GET | `/api/v1/ranking/posts/daily` | 文章日榜 |
| GET | `/api/v1/ranking/posts/weekly` | 文章周榜 |
| GET | `/api/v1/ranking/posts/monthly` | 文章月榜 |
| GET | `/api/v1/ranking/creators/hot` | 创作者热度排行 |
| GET | `/api/v1/ranking/topics/hot` | 话题热度排行 |

### 3.3 创作者排行聚合逻辑

- **聚合公式**：创作者影响力分数 = 该创作者所有文章 final_score 之和（取 Top 50 篇文章，避免大量低分文章稀释排名区分度）
- **计算时机**：在文章热度快照生成完成后，二次聚合生成创作者排行快照。不独立维护增量，始终从文章快照数据派生
- **数据关联**：通过 MongoDB 文章热度文档中的 `authorId` 字段关联，快照生成时按 `authorId` 分组聚合

### 3.4 话题排行聚合逻辑

- **聚合公式**：话题热度分数 = 该话题下所有关联文章 final_score 之和（取 Top 100 篇文章）
- **文章-话题关联**：文章发布时携带 `topicIds`（标签列表），ranking 服务在消费互动事件时同步记录 `postId -> topicIds` 映射关系到 MongoDB
- **计算时机**：与创作者排行相同，在文章快照生成后二次聚合。按 `topicId` 分组，对关联文章的 final_score 求和
- **注意**：一篇文章可关联多个话题，其 final_score 会被计入每个关联话题的总分

### 3.2 请求/响应示例

```
GET /api/v1/ranking/posts/hot?page=1&size=20

Response 200:
{
  "code": 0,
  "data": {
    "total": 100,
    "items": [
      {
        "postId": 10001,
        "title": "Spring Boot 实战指南",
        "authorName": "张三",
        "score": 8523.6,
        "viewCount": 5200,
        "likeCount": 320,
        "commentCount": 85,
        "favoriteCount": 120,
        "rank": 1
      }
    ]
  }
}
```

#### rank 字段语义说明

- `rank` 为快照生成时预计算的全局排名（从 1 开始），存储在 MongoDB 快照文档的每个 item 中
- 分页查询时，rank 值反映的是全局排名而非页内序号。例如 page=2, size=20 时，第一条的 rank 为 21
- rank 不保证连续性：若中间有文章被删除/下架过滤，rank 可能出现间隔（如 1, 2, 4）。前端展示时建议使用 `(page-1)*size + index + 1` 作为展示序号

#### 错误码定义

| 错误码 | HTTP 状态码 | 说明 |
|--------|-----------|------|
| 0 | 200 | 成功 |
| 40001 | 400 | 分页参数非法（page < 1 或 size > 100） |
| 40002 | 400 | 排行类型不存在 |
| 42900 | 429 | 请求被限流，稍后重试 |
| 50001 | 500 | 排行数据查询内部错误 |
| 50003 | 503 | 服务降级中（Redis 和 MongoDB 均不可用） |

## 4. 数据模型

### 4.1 MongoDB 排行快照文档

```json
{
  "_id": "ObjectId",
  "type": "POST_HOT | POST_DAILY | POST_WEEKLY | POST_MONTHLY | CREATOR | TOPIC",
  "period": "2026-02-26",
  "items": [
    {
      "targetId": 10001,
      "score": 8523.6,
      "rank": 1,
      "metadata": { "title": "...", "authorName": "..." }
    }
  ],
  "generatedAt": "ISODate",
  "expireAt": "ISODate"
}
```

#### metadata 一致性策略

- **快照中的 metadata（title、authorName）为冗余缓存**，权威数据源在 content-service
- **更新机制**：监听 content-service 发出的文章信息变更事件（标题修改）和用户信息变更事件（改名），异步更新 MongoDB 快照中对应排行项的 metadata
- **最终一致性**：metadata 不要求强一致，允许短暂延迟（下次快照生成时会从 content-service 批量拉取最新信息覆盖）
- **降级方案**：若 content-service 不可用，快照生成时沿用上一版本的 metadata，不阻塞快照生成流程

- 索引：`{ type: 1, period: -1 }` 复合索引，加速按类型+时间查询
- TTL：通过 `expireAt` 字段自动清理过期快照（日榜保留 30 天，周榜 90 天，月榜 365 天）
- **快照数据量上限**：每个快照文档只存储 Top 1000 排行项。按每个 item 约 200 字节估算，单文档约 200KB，远低于 MongoDB 16MB 文档上限
- **超大排行需求**：若未来需要存储超过 Top 1000 的完整排行数据，改为每个排行项一个独立文档，通过 `{ type: 1, period: 1, rank: 1 }` 复合索引查询，避免单文档膨胀

### 4.2 Redis 缓存 Key 设计

| Key 模式 | 数据结构 | TTL | 说明 |
|----------|---------|-----|------|
| `ranking:posts:hot` | Sorted Set | 无 TTL（由快照全量刷新管理） | 文章热度总榜 |
| `ranking:posts:daily:{date}` | Sorted Set | 2d | 文章日榜 |
| `ranking:posts:weekly:{week}` | Sorted Set | 7d | 文章周榜 |
| `ranking:posts:monthly:{year}:{month}` | Sorted Set | 62d | 文章月榜 |
| `ranking:creators:hot` | Sorted Set | 无 TTL | 创作者总榜 |
| `ranking:topics:hot` | Sorted Set | 无 TTL | 话题总榜 |

#### 总榜更新策略与淘汰机制

- **更新方式**：总榜不再使用增量 ZINCRBY，改为每次快照生成时全量刷新（DELETE + ZADD 批量写入 Top N）。原因：增量 ZINCRBY 与 TTL 过期互相矛盾，且无法在增量模式下应用时间衰减
- **成员淘汰**：快照生成时只将 Top 10000 的文章写入 Redis Sorted Set，超出部分不写入。已存在但不在新快照中的成员通过全量替换自动淘汰
- **全量刷新原子性**：当前实现使用临时 key 批量写入后 `RENAME` 覆盖目标 key，并在覆盖后恢复 TTL，避免 DELETE 后 ZADD 前的短暂空窗期
- **内存容量预估**：单个 Sorted Set 成员约 50 字节（member + score），Top 10000 约 500KB，6 个排行维度总计约 3MB，Redis 内存占用可控

## 5. 非功能设计

- 缓存策略：查询优先读 Redis，miss 时回源 MongoDB 并回填缓存
  - **防击穿**：缓存 miss 时使用 Redisson 分布式锁（`ranking:lock:load:{type}:{period}`），同一排行维度只允许一个请求回源 MongoDB，其余请求等待锁释放后读取已回填的缓存，等待超时 3s 后降级返回空结果
  - **防穿透**：MongoDB 中也不存在的排行类型（如新维度上线初期），缓存空结果并设置短 TTL（30s），避免无效查询持续打到 DB
  - **防雪崩**：各排行维度的缓存 TTL 增加随机偏移量（±60s），避免多个缓存同时过期；总榜采用逻辑过期策略——缓存不设物理 TTL，由快照生成任务主动刷新，查询时检查逻辑过期时间戳，过期则触发异步刷新并返回旧数据
- 聚合作业与快照刷新：
  - **聚合作业**：`RankingInboxScheduler` 按 `ranking.inbox.scan-interval` 扫描 inbox
  - **快照刷新**：`RankingSnapshotScheduler` 按 `ranking.snapshot.refresh-interval` 重建当前 Redis 快照
  - **分布式锁**：两个调度器都通过 `DistributedLockExecutor` 使用 Redis 锁，确保多实例下同一时刻只有一个实例执行
- 限流：Sentinel 按接口维度限流，单接口 500 QPS
- **文章状态过滤**：
  - 监听 content-service 发出的文章状态变更事件（删除/下架/封禁），收到后从 Redis Sorted Set 中 `ZREM` 对应成员，并标记 MongoDB 热度文档中该文章为 `status=INACTIVE`
  - 快照生成时过滤 `status=INACTIVE` 的文章，不写入快照
  - 文章恢复上架时，重新标记为 `status=ACTIVE`，下次快照生成时自动恢复排行
- 可观测性：
  - 当前已落地指标：`ranking.view.dedup`
  - 建议重点监控：inbox `NEW/PROCESSING/FAILED` 数量、聚合作业耗时、快照刷新耗时、Redis key 大小
- 优雅停机：不再依赖 JVM 内存缓冲刷写；重点保证未完成 inbox 记录在租约超时后可被重新接管
